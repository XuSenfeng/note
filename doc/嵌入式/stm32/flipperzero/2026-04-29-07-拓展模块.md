# 拓展模块

flipperzero和拓展的模块通信的时候有一个标准的协议Expansion Module Protocol, 用于定义 Flipper Zero（主机）与第三方 GPIO 扩展模块（设备）之间的**串口同步通信规则**，是扩展硬件开发的核心对接标准

+   **主机 (Host)**：Flipper Zero，提供服务与响应
+   **设备 (Device)**：第三方扩展模块，发起通信
+   **通信超时 Tto**：250ms，无交互即断连
+   **波特率切换死区 Tdt**：25ms，切速后禁通信
+   **RPC**：基于 Protobuf 的远程调用，协议可与其对接

## 核心功能

1.   自动检测扩展模块
2.   动态波特率协商
3.   基础异或校验错误检测
4.   严格请求 - 响应交互
5.   兼容 Flipper RPC 协议

## 硬件接口

| UART   | Tx 引脚 | Rx 引脚 |
| ------ | ------- | ------- |
| USART  | 13      | 14      |
| LPUART | 15      | 16      |

## 软件接口

所有帧统一格式：**1 字节头部 + 可变长度内容 + 1 字节异或校验和**

校验规则：帧内除校验位外所有字节逐位异或，初始值 0

### 核心帧

#### 心跳帧 HEARTBEAT (0x01)

无内容，仅头部 + 校验，用于保活；超时未收则重置连接

#### 状态帧 STATUS (0x02) 

内容为 1 字节错误码，每帧必回

-   0x00：成功
-   0x01：未知错误
-   0x02：波特率不支持

#### 波特率帧 BAUD RATE (0x03)

内容 4 字节速率值；初始必用 9600 波特，协商成功前不可发其他帧, 模块发送的第一条消息必须是波特率帧

协调成功的时候, 返回一个OK状态位, 否则是0x2

#### 控制帧 CONTROL (0x04)

内容 1 字节命令，管理会话与电源

-   0x00：启动 RPC 会话
-   0x01：停止 RPC 会话
-   0x02：开启 GPIO 5V OTG
-   0x03：关闭 GPIO 5V OTG

#### 数据帧 DATA (0x05)

最大承载 64 字节用户数据；超 64 字节需分包；RPC 数据走此帧传输

| Data size (1 byte) | Data (0 to 64 bytes) |
| ------------------ | -------------------- |
| 0x00 ... 0x40      | Arbitrary data       |

### 通信流程

想要 Flipper（主机）能检测到外接模块，**必须先开启「扩展模块监听功能」**：

1.  **图形界面开启**：设置 → 扩展模块 → 选择对应的监听串口（Listen UART）
2.  **代码开启**：调用函数 `expansion_enable()`
3.  **关闭功能**：通过上述 GUI 或调用 `expansion_disable()` → 所有通信立即停止，Flipper 再也无法检测任何外接模块

**通信永远由「外接模块」主动发起**，Flipper 只做响应, 核心超时参数：`Tto = 250ms`（无交互就断连）；`Tdt = 25ms`（切换波特率后的等待时间）

```bash
  MODULE（外接模块）       |            FLIPPER（主机）
-----------------------------+---------------------------
                             |        (Start) 【初始状态：等待连接】
Pull down RX                -->  【模块硬件动作：短暂拉低RX引脚】
                            <--       Heartbeat 【Flipper必须回复：心跳帧=我准备好了】
Baud Rate                   -->  【模块必须在250ms内发送：波特率协商帧】
                            <--       Status [OK | Error] 【Flipper回复：成功/失败】
                             |
(模块切换波特率，并等待25ms)   |        (Flipper同步切换波特率)
                             |
Control [Start RPC]         -->  【模块发送：启动RPC会话命令】
                            <--       Status [OK | Error] 【Flipper确认】
-----------------------------+--------------------------- (1) 【单包数据传输】
Data [RPC请求]              -->  【模块发送RPC数据】
                            <--       Status [OK | Error] 【Flipper先确认收到】
                            <--       Data [RPC响应] 【Flipper返回处理结果】
Status [OK | Error]         -->  【模块必须回复确认帧】
-----------------------------+--------------------------- (2) 【大包数据分包传输】
Data [RPC请求 第1部分]       -->  【数据超64字节，拆分发送】
                            <--       Status [OK | Error]
Data [RPC请求 第2部分]       -->
                            <--       Status [OK | Error]
Data [RPC请求 第3部分]       -->
                            <--       Status [OK | Error]
                            <--       Data [RPC响应] 【所有分包收完，返回响应】
Status [OK | Error]         -->  【模块最终确认】
-----------------------------+--------------------------- (3) 【心跳保活】
Heartbeat                   -->  【无数据时，模块定期发心跳】
                            <--       Heartbeat 【Flipper必回心跳】
Heartbeat                   -->
                            <--       Heartbeat
-----------------------------+---------------------------
Control [Stop RPC]          -->  【模块主动结束会话】
                            <--       Status [OK | Error] 【Flipper确认】
(模块断开连接)               |
                             |        (250ms无活动，自动回到初始状态)
```

>   模块**必须**用 `STATUS` 帧，确认所有 Flipper 发来的响应数据帧。
>
>   RPC 数据**超过 64 字节**时，必须拆分成多个小数据帧发送, **每发送一个分包**，都必须等待 Flipper 回复 `STATUS` 确认后，才能发下一个
>
>   模块没有数据要传输时，**必须定期发送心跳帧**，且发送间隔 **小于 250ms**
