# 复位烧录升级

## 复位模式
### 硬件复位

当开发板实在无法使用的时候, 可以同时按下`LEFT`以及`BACK`按键
按键 → DD8(NC7SZ32M5X 或门)→ DD1(STM32WB55 主控),直接把 MCU 的复位线拉低。调试的时候调试模块可能导致内部锁死复位引脚

### 硬件断电复位

断开 USB 和所有外部供电 → 长按 `BACK` 30 秒 → 松开
原理:经过充电芯片 DD6(bq25896),把 SYS 电源线先关后开

### 软件UFD

- 开机时按住 `LEFT`,进入 **Flipper 自带 bootloader** 的 DFU 模式。
- **失效场景**:Flipper bootloader 损坏或缺失。

### 硬件UFD

- 开机时按住 `OK`,进入 **ST 原厂 bootloader** 的 DFU 模式。
- **失效场景**:Option Bytes 损坏,或被设置成忽略 `OK` 键

> 关键区分:**Software DFU 用 Flipper 的引导程序,Hardware DFU 用 STM32 芯片内置的引导程序**。前者依赖固件完好,后者是芯片级的最后手段


## UFD组合

**DFU = Device Firmware Update**,是 **USB-IF 官方定义的一个标准 USB 设备类**。它的作用是:让一块单片机通过普通 USB 线就能被重新刷写固件,而不需要专用的烧录器(如 ST-Link/SWD)。

- **Hardware DFU** → STM32 芯片**出厂内置的 ROM bootloader**(地址 `0x0` 系统存储区)。芯片级,永远存在,刷不坏。
- **Software DFU** → Flipper 自己实现的引导逻辑, 绘制蓝屏等显示处理。最后还是进入硬件的DFU里面去


### 硬件复位+软件DFU
- Press `LEFT` and `BACK` and hold for a couple of seconds
- Release `BACK`
- Device will enter DFU with an indication (Blue LED + DFU Screen)
- Release `LEFT`

### 硬件复位+硬件DFU

- Press `LEFT`, `BACK` and `OK` and hold for a couple of seconds
- Release `BACK` and `LEFT`
- The device will enter DFU without an indication

### 电源复位+软件DFU

- Disconnect the USB and any external power supplies
- Press `BACK` and `LEFT` for 30 seconds
- Release `BACK`
- The device will enter DFU with an indication (Blue LED + DFU Screen)
- Release `LEFT`
- Plug in the USB

### 电源复位+硬件DFU

- Disconnect the USB and any external power supplies
- Press `BACK` and `OK` and hold for 30 seconds
- Release `BACK` and `OK`
- The device will enter DFU without indication
- Plug in the USB

## 软件代码实现

### 进入DFU模式
```c
// 获取启动模式
FuriHalRtcBootMode boot_mode = furi_hal_rtc_get_boot_mode();
if(boot_mode == FuriHalRtcBootModeDfu || !furi_hal_gpio_read(&gpio_button_left)) {
	furi_hal_light_sequence("rgb WB");
	furi_hal_rtc_set_boot_mode(FuriHalRtcBootModeNormal);
	flipper_boot_dfu_exec();  // 处理ufd模式
	furi_hal_power_reset();
}
```

```c
void flipper_boot_dfu_exec(void) {
    flipper_boot_dfu_show_splash();              // 画 "Update & Recovery Mode / DFU Started" 蓝屏
    WRITE_REG(FLASH->SR, FLASH_SR_OPTVERR);      // 清 STM32 勘误表 2.2.9 的误置标志
    furi_hal_deinit_early();                     // 关闭外设/时钟,恢复干净状态
    LL_SYSCFG_SetRemapMemory(LL_SYSCFG_REMAP_SYSTEMFLASH); // 把地址 0x0 重映射到系统 bootloader
    furi_hal_switch(0x0);                         // 跳转到 0x0 = ST 内置 ROM bootloader
}
```

把内存 `0x0` 重映射到 **STM32 的系统 Flash(ROM bootloader)**,然后跳过去。跳转之后,就由芯片内置的 USB DFU 类接管,PC 上的 dfu-util / STM32CubeProgrammer 就能识别它并刷固件了。

## 升级

可以解析SD卡里面的dfu文件进行烧录升级 `update_task_write_dfu`

### CRC校验
进行CRC校验，DFU 文件末尾 4 字节(`DfuSuffix.dwCRC`)存的是"前面所有内容 CRC32 的取反"。所以**把整个文件(含这 4 字节)一起算 CRC32,结果必须等于 `0xFFFFFFFF`**。等于就说明文件一字节没坏。`dfu_file_validate_crc`
### 头尾校验 
读 `DfuPrefix`(文件头)和 `DfuSuffix`(文件尾),检查:
- 文件头签名必须是 `"DfuSe"`,版本为 1;
- `DFUImageSize` 必须等于实际数据区大小(防截断);
- 文件尾 `bcdDFU == 0x011A`、`bLength` 正确;
- **关键的设备匹配**——文件尾里的 VID/PID/设备码必须和本机一致
不匹配直接返回 0 → 拒绝刷写。**这就是防止把别的设备/型号固件刷进来的保险。** 返回值是 target 数量。
### 逐页烧录

DFU 文件的数据结构是三层嵌套:**文件 → 多个 Target → 每个 Target 多个 Image Element → 每个Element是一段连续内存映像**。代码就按这个层级遍历:

1. 读每个 `TargetPrefix`;
2. 读每个 `ImageElementHeader`(含**烧写起始地址** `dwElementAddress` 和大小);
3. 进入 `dfu_file_perform_task_for_update_pages()` 真正烧写:
    - **地址必须页对齐**,否则失败;
    - 调 `address_cb`(`check_address_boundaries`)验证地址落在可写 Flash 范围内,**越界的 element 直接跳过(Skipped)**——这保护了 bootloader 等保留区不被覆盖;
    - 按 Flash 页大小循环读取,逐页调 `task_cb` = `update_task_flash_program_page()` → `furi_hal_flash_program_page()` 写入。
### 回读校验

在函数`page_task_compare_flash`里面查看是否写入的正确

### 生成升级包

SD 卡 OTA 走的需要的是一个**完整更新包**:里面有 `update.fuf` 清单、`.dfu`、蓝牙 stack、资源等,通常打包成 `flipper-z-f7-update-local.tgz`。代码读的是 `manifest->firmware_dfu_image`——**它从清单里找 `.dfu`,而不是单独一个文件**。

```bash
./fbt updater_package      # 生成 dist/f7-D/.../*.tgz 完整更新包
```

```bash
# 1. 解压 zip,只取 f7-update-local 这一个文件夹
unzip dist/f7-D/flipper-z-f7-sdk-local.zip "f7-update-local/*"

# 2. 把整个文件夹拷到 SD 卡 /update/f7-update-local/
#    (用 qFlipper 的文件管理,或拔卡用读卡器)

# 3. 在 Flipper 上:Settings → System → Install from SD card
#    选中 update.fuf,设备会自动重启 → 烧主固件 + radio stack + 资源
```

这条路比 USB DFU 更"完整":会一并刷蓝牙协处理器、解压资源、装开机画面——就是 `update_task_alloc` → `backup_restore` → 重启 → `flash_writer` 的全流程都跑一遍。
