# APP开发

FAP (Flipper App Package) 支持将应用构建为 FAP 文件。FAP 本质上是附带了额外元数据和资源的 `.elf` 可执行文件。

## 创建app

要将你的应用构建为 FAP，请先在 `applications_user` 中创建一个包含应用源码的文件夹，然后像编写普通内置应用一样完成代码。接着配置它的 `application.fam` 清单文件，并将其 `apptype` 设置为 `FlipperAppType.EXTERNAL`

### 默认创建

创建应用可以使用ufbt

```bash
ufbt vscode_dist create APPID=myapp LANG_SERVER=clangd
ufbt vscode_dist LANG_SERVER=clangd # 使用clangd的配置
```

## 编译命令

```bash
# 编译其中一个App
./fbt fap_{APPID}
# 编译并且在开发板运行
./fbt launch APPSRC=applications_user/path/to/app
# 编译同步到开发板
./fbt build APPSRC=applications_user/path/to/app
# 编译所有
./fbt faps or 
# 目标部署到 `dist` 文件夹中
./fbt fap_dist
```

### application.fam

```c
App(
    appid="gpio_control",
    name="Example: GPIO Control",
    apptype=FlipperAppType.EXTERNAL,
    entry_point="gpio_control_main",
    requires=["gui"],
    stack_size=1 * 1024,
    fap_icon="gpio_control.png",
    fap_category="GPIO",  # 设置分类文件夹
)
```

Flipper Zero 固件中的所有组件，包括服务、用户应用和系统设置，都是独立开发的。每个组件都有一个名为 `application.fam` 的构建系统清单文件，它定义了该组件的基本属性以及它与系统其他部分之间的关系。
在构建固件时，`fbt` 会收集所有应用清单并处理它们的依赖关系。随后，它只会构建当前构建配置中引用到的那些组件。
只有两个参数是必填的：**appid** 和 **apptype**。其他参数都是可选的，并且可能只对某些应用类型有意义。
同一个`.fam`文件里面可以有多个App
#### 参数
##### 基础参数
- **appid**：字符串，构建系统中的应用 ID。用于指定在构建配置中包含哪些应用，以及解析依赖关系和冲突关系。
- **apptype**：`FlipperAppType.*` 枚举成员。有效值如下：

| 枚举成员         | 固件组件类型                                                                                                                   |
| ------------ | ------------------------------------------------------------------------------------------------------------------------ |
| SERVICE      | 系统服务，在早期启动阶段创建                                                                                                           |
| SYSTEM       | 不会显示在任何菜单中的应用。可以由其他应用或通过 CLI 启动                                                                                          |
| APP          | 主菜单中的常规应用, 链接在固件里面                                                                                                       |
| PLUGIN       | 作为固件一部分构建，并放入 Plugins 菜单中的应用                                                                                             |
| DEBUG        | 仅在启用调试模式时显示在 Debug 菜单中的应用                                                                                                |
| ARCHIVE      | 唯一的 Archive 应用                                                                                                           |
| SETTINGS     | 放入系统设置菜单中的应用                                                                                                             |
| STARTUP      | 在系统启动时运行的回调函数。不定义独立应用                                                                                                    |
| EXTERNAL     | 构建为 `.fap` 插件的应用, 总是被部署                                                                                                  |
| MENUEXTERNAL | 构建为 `.fap` 插件的应用, 跟随 app set 被部署, 用这个类型的全是**核心功能应用**：`gpio`、`nfc`、`subghz`、`infrared`、`lfrfid`、`ibutton`、`bad_usb`、`u2f` |
| METAPACKAGE  | 不定义任何可执行代码，用于声明依赖和应用包                                                                                                    |

- **name**：在菜单中显示的名称。
- **entry_point**：作为应用入口点使用的 C 函数。注意，C++ 函数名会经过名称修饰，因此若要将其用作入口点，需要用 `extern "C"` 包裹。
- **flags**：系统应用的内部标志。不要使用。
- **cdefines**：当当前应用被包含在活动构建配置中时，为其他应用全局声明的 C 预处理器定义。**对于外部应用**：指定的定义会在构建应用自身时使用。
- **requires**：应用 ID 列表。当当前应用出现在待构建应用列表中时，这些应用也会被加入构建配置。
- **conflicts**：与当前应用冲突的应用 ID 列表。如果在最终生成的应用列表中发现其中任意一项，`fbt` 会中止固件构建过程。
- **provides**：功能上与 **requires** 字段相同。
- **stack_size**：应用启动时分配的栈大小，单位为字节。注意，如果分配的栈太小，应用运行时可能因栈溢出导致系统崩溃；如果分配过大，则会减少应用处理数据时可用的堆内存大小。_注意：你可以使用 `top` 和 `free` CLI 命令分析应用的内存使用情况。_
- **icon**：从内置资源中选择的动态图标名称，在应用作为固件一部分构建时使用。
- **order**：应用在所属分组中的排序顺序。数值越小，条目越靠前。_用于启动钩子和菜单项排序。_
- **sdk_headers**：从当前应用代码中选出的 C 头文件列表，这些头文件会被纳入外部应用的 API 定义。
- **targets**：与当前应用兼容的目标名称字符串列表。如果未指定，则该应用会为所有目标构建。默认值为 `["all"]`。
- **resources**：应用源码目录中的某个文件夹名称，用于打包该应用的 SD 卡资源。只有当应用被包含进构建配置时，这些资源才会被使用。默认值为 `""`，表示不打包任何资源。
##### 外部应用参数

- **sources**：字符串列表，用作收集应用文件夹中源码的文件名匹配规则。默认值 `["*.c*"]` 会包含 C 和 C++ 源文件。应用不能将 `"lib"` 文件夹用于自身源码，因为该目录保留给 **fap_private_libs**。以 `"!"` 开头的路径会从源码列表中排除。它们也可以包含通配符和目录名。例如，`["*.c*", "!plugins"]` 会包含应用目录中的所有 C 和 C++ 源文件，但排除 `plugins`（以及 `lib`）目录中的文件。不包含通配符（`*`、`?`）的路径，在包含和排除时都会被视为完整的字面路径。
- **fap_version**：字符串，应用版本。默认值为 `"0.1"`。你也可以使用 `(x,y)` 形式的两个数字元组来指定版本。还可以添加更多以点分隔的版本段，例如补丁号，但构建出的 `.fap` 中只会保存主版本号和次版本号。
- **fap_icon**：一个 `.png` 文件名，1 位色深，尺寸为 10x10 像素，用于嵌入到 `.fap` 文件中。
- **fap_libs**：需要额外链接到应用的库列表。它可让应用访问未作为主固件一部分导出的额外函数，但代价是 `.fap` 文件体积和 RAM 占用增加。
- **fap_category**：字符串，可为空。应用子分类，同时也决定 FAP 在文件系统中 `apps` 文件夹下的路径。
- **fap_description**：字符串，可为空。应用的简短描述。
- **fap_author**：字符串，可为空。应用作者。
- **fap_weburl**：字符串，可为空。应用主页。
- **fap_icon_assets**：字符串。如果存在，则定义一个文件夹名称，用于收集该应用的图像资源。这些图像会被预处理并与应用一同构建。
- **fap_extbuild**：支持通过外部工具构建应用源码的一部分。它包含一个 `ExtFile(path="file name", command="shell command")` 定义列表。`fbt` 会针对列表中的每个文件运行指定命令。
- **fal_embedded**：布尔值，默认 `False`。仅适用于 `PLUGIN` 类型。如果为 `True`，该插件会作为资源嵌入宿主应用的 `.fap` 文件，并在启动时解压到 `apps_assets/APPID` 文件夹。这使得插件可以作为宿主应用的一部分进行分发。
  
注意，这些命令是在固件根目录下执行的，所有中间文件都必须放在应用的临时构建目录中。为此，你可以使用 `fbt` 提供的模式展开：`${FAP_WORK_DIR}` 会被替换为应用临时构建目录的路径，`${FAP_SRC_DIR}` 会被替换为应用源码目录的路径。你也可以使用 `fbt` 内部定义的其他变量。

```python
sources=["target/thumbv7em-none-eabihf/release/libhello_rust.a"],
fap_extbuild=(
	ExtFile(
		path="${FAP_WORK_DIR}/target/thumbv7em-none-eabihf/release/libhello_rust.a",
		command="cargo build --release --verbose --target thumbv7em-none-eabihf --target-dir ${FAP_WORK_DIR}/target --manifest-path ${FAP_SRC_DIR}/Cargo.toml",
	),
),
```


- **fap_private_libs**：与应用源码一起分发的附加库列表。这些库会作为应用构建过程的一部分一起编译。
  库源码必须放在应用源码目录下 `lib` 文件夹的某个子目录中。
  每个库都通过调用 `Lib()` 函数定义，并接受以下参数：
>   **name**：库文件夹的名称。必填。
  > **fap_include_paths**：需要加入父级 fap 头文件搜索路径列表的、相对于库目录的路径列表。默认值为 `["."]`，表示库源码根目录。
  > **sources**：用于收集该库源码文件的文件名匹配规则列表。路径相对于库源码根目录。默认值为 `["*.c*"]`。
  > **cflags**：构建该库时使用的附加编译器参数列表。默认值为 `[]`。
  > **cdefines**：构建该库时使用的附加预处理器定义列表。默认值为 `[]`。
  > **cincludes**：构建该库时使用的附加头文件搜索路径列表。路径相对于应用根目录。它可用于为该库代码提供额外的外部搜索路径，例如配置头文件。默认值为 `[]`。
  
```python
fap_private_libs=[
		Lib(
			name="mbedtls",
			fap_include_paths=["include"],
			sources=[
				"library/des.c",
				"library/sha1.c",
				"library/platform_util.c",
			],
			cdefines=["MBEDTLS_ERROR_C"],
		),
		Lib(
			name="loclass",
			cflags=["-Wno-error"],
		),
	],
```

对于上面的代码片段，`fbt` 会构建 2 个库：一个来自 `lib/mbedtls` 目录中的源码，另一个来自 `lib/loclass` 目录中的源码。对于 `mbedtls` 库，`fbt` 会将 `lib/mbedtls/include` 添加到该应用的头文件搜索路径列表中，并且只编译 `sources` 列表中指定的文件。此外，`fbt` 还会为 `mbedtls` 源码启用 `MBEDTLS_ERROR_C` 预处理器定义。

对于 `loclass` 库，`fbt` 会将 `lib/loclass` 添加到该应用的头文件搜索路径列表中，并构建该目录中的全部源码。同时，`fbt` 会对 `loclass` 库关闭“将编译器警告视为错误”的设置，这在编译大型第三方代码库时会很有用。
  
这两个库都会与应用一起链接。

## 资源文件

FAP 可以包含静态图像和动态图像作为私有资源。这些资源会在构建应用源码时自动一并编译，并且可以像主固件中的资源一样被引用。
要使用该功能，请将图像放到应用目录下的某个子文件夹中，然后在应用 manifest 的 `fap_icon_assets` 字段中引用该文件夹。更多细节请参见 [Flipper App Manifests](AppManifests.md)。

要在应用中使用这些资源，请在源码中加入 `#include "{APPID}_icons.h"`，其中 `{APPID}` 是应用 manifest 中的 `appid` 字段值。之后，你就可以像使用主固件 `assets_icons.h` 中的资源一样，使用应用资源里的全部图标。

图像和动态图标应遵循与主固件资源相同的[[2026-5-4-11-GUI模块]]。

## 调试 FAP
  
`fbt` 包含一个供 gdb-py 使用的脚本 `debug/flipperapps.py`，用于为 FAP 提供调试支持。它会由 `fbt` 和默认的 VS Code 调试配置自动加载。
  
借助它，你可以像调试主固件的一部分那样调试 FAP，例如查看变量、设置断点、单步执行代码等。
  
如果调试会话处于活动状态，固件会在将 FAP 加载到内存之后、执行其中任何代码之前主动触发一个断点。这样你就可以在 FAP 代码中设置断点。需要注意的是，在 FAP 加载之前设置的断点，可能需要在 FAP 实际加载后重新设置，因为在加载前调试器无法知道 FAP 代码的确切地址。

### 配置调试环境
  
调试支持脚本会在最新的固件构建目录 `build/latest` 中查找调试信息。当你针对所选配置运行 `./fbt` 时，`fbt` 会将该目录符号链接到最新的固件配置（Debug 或 Release）构建目录。详见 [fbt docs](fbt.md)。
要调试 FAP，请执行以下步骤：
  
1. 使用 `./fbt` 构建固件
2. 使用 `./fbt flash` 将其烧录到设备
3. 构建你的 FAP 并在 Flipper 上运行它
之后，你就可以通过 `./fbt debug` 或 VS Code 将调试器附加到目标 MCU，并使用全部调试功能。
**重要：** 固件与应用的构建类型（debug/release）必须一致，并且对应的固件目录必须链接为 `build/latest`。否则调试将无法正常工作。

## Flipper 如何从 SD 卡运行应用
  
Flipper 的 MCU 无法直接从外部存储执行代码，因此必须先将代码复制到 RAM。这个过程由 App Loader 完成，它负责从 SD 卡加载 FAP、校验其完整性和兼容性、将其复制到 RAM，并根据新的加载位置进行必要调整。
  
由于 FAP 必须先加载到 RAM 才能执行，因此与作为固件一部分、直接从 flash 运行同一个应用相比，可用于堆分配的 RAM 会减少。需要注意的是，占用的 RAM 量小于 FAP 文件总大小，因为只有代码段和数据段会被分配到内存中，而 FAP 文件还包含一些仅在应用加载时使用的额外信息。
  
应用是针对特定 API 版本构建的。该版本属于硬件目标定义的一部分，包含主版本号和次版本号。App Loader 会检查应用的 API 主版本号是否与固件的 API 主版本号一致。
  
App Loader 会为应用分配内存，并将其复制到 RAM 中，同时处理重定位，并通过**符号表**为导入符号提供具体地址。随后它会启动该应用。

## API 版本管理
  
并非固件的所有部分都对外部应用开放。可用函数和变量的一个子集定义在 `api_symbols.csv` 文件中，该文件是 `targets/` 目录下固件目标定义的一部分。
  
`fbt` 对 API 使用语义化版本管理。出现破坏性变更时，会增加主版本号；增加新特性时，会增加次版本号。
  
破坏性变更包括：
  
- 删除某个函数或全局变量
- 修改函数签名
  
API 版本管理大部分由 `fbt` 自动完成。当重新构建固件时，`fbt` 会检查从 `SDK_HEADERS` 收集到的头文件中，对外暴露的 API 是否发生变化。如果有变化，它会停止构建、调整 API 版本，并要求用户检查 `.csv` 文件中的变更。新增条目会被标记为 "`?`"，用户需要将其改为 "`+`" 以对 FAP 开放，或改为 "`-`" 以表示不可用。
  
在所有 "`?`" 条目都被修改为 "`+`" 或 "`-`" 之前，`fbt` 不允许构建固件。
  
**注意：** `fbt` 会自动管理 API 版本。唯一允许且必须手动增加 API 主版本号的情况，是将现有的 "`+`" 条目改为 "`-`" 时。
  
### 符号表

符号表是由固件导出、可供外部应用使用的一组符号列表。它由 `fbt` 基于 API symbols 文件生成，并由 App Loader 用于解析导入符号的地址。它会作为 `fap_loader` 应用的一部分被构建。
  
`fbt` 还会检查所有导入符号是否都存在于符号表中。如果缺少任何符号，它会发出警告并列出这些符号。在符号表提供所有所需符号之前，该应用将无法在设备上运行。
