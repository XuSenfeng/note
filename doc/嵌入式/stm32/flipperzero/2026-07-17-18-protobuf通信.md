# protobuf通信

标准的协议可以查看[[2026-07-17-01-Protocol-Buffers]]

Flipper 固件用 protobuf 做 RPC 通信，但用的不是标准的 protobuf 库，而是 **nanopb**（专为嵌入式/单片机设计的轻量实现）。

nanopb 的一个核心限制是：默认情况下它需要给每个字段分配**固定大小的静态内存**。比如一个 `string key = 1;` 字段，nanopb 会默认生成一个固定长度的 `char key[N];` 数组，你必须提前告诉它 N 是多少

## .option文件

`.options` 文件就是用来给这些字段**附加生成代码时的配置**，而不用去改 `.proto` 文件本身


```protobuf
PB_Storage.File.name                          type:FT_POINTER
PB_Storage.File.data                          type:FT_POINTER
PB_Storage.InfoRequest.path                   type:FT_POINTER
PB_Storage.TimestampRequest.path              type:FT_POINTER
PB_Storage.StatRequest.path                   type:FT_POINTER
PB_Storage.ListRequest.path                   type:FT_POINTER
PB_Storage.ReadRequest.path                   type:FT_POINTER
PB_Storage.WriteRequest.path                  type:FT_POINTER
PB_Storage.DeleteRequest.path                 type:FT_POINTER
PB_Storage.MkdirRequest.path                  type:FT_POINTER
PB_Storage.Md5sumRequest.path                 type:FT_POINTER
PB_Storage.RenameRequest.old_path             type:FT_POINTER
PB_Storage.RenameRequest.new_path             type:FT_POINTER
PB_Storage.BackupCreateRequest.archive_path   type:FT_POINTER
PB_Storage.BackupRestoreRequest.archive_path  type:FT_POINTER
PB_Storage.TarExtractRequest.tar_path         type:FT_POINTER
PB_Storage.TarExtractRequest.out_path         type:FT_POINTER

PB_Storage.ListResponse.file                  max_count:8

// not affected by nanopb, so server & client should keep in mind these max values
PB_Storage.File.data                          max_size:512
PB_Storage.Md5sumResponse.md5sum              max_length:32
PB_Storage.File.md5sum                        max_length:32
PB_Storage.*.path                             max_length:255
```


### 基本行格式

每一行的结构是：

```
<字段选择器>    <选项名>:<值>  [<选项名>:<值> ...]
```

- 用**空白**（空格/Tab）分隔选择器和选项，具体对齐几个空格无所谓，纯粹为了好看。
- 一行可以写多个选项，空格分开，例如 `foo.bar type:FT_POINTER max_size:64`。
- 空行被忽略。
- `//` 是注释（见第 21 行），nanopb 的 options 解析器支持 C 风格注释。

### 字段选择器

选择器用**点分路径**定位到 `.proto` 里的具体字段：

```
PB_Storage . File . name
  消息包名     子消息   字段名
```

- 层级对应 proto 里的嵌套 message。`PB_Storage.File.name` = `PB_Storage` 消息里嵌套的 `File` 子消息里的 `name` 字段。
- **支持通配符 `*`**，这是关键特性。看第 25 行：

```
PB_Storage.*.path        max_length:255
```

意思是：`PB_Storage` 下**所有**子消息里名叫 `path` 的字段，最大长度都设成 255。这样就不用像第 3–11 行那样一个个写 `max_length` 了。匹配规则是从最具体到通配，nanopb 会对每个字段找匹配的规则应用。

### 常用选项详解

#### `type:FT_POINTER`

控制字段的**内存分配方式**，这是 nanopb 最核心的 option。可选值：

| 值             | 含义                                                   |
| ------------- | ---------------------------------------------------- |
| `FT_STATIC`   | 默认。生成固定大小的内联数组/结构，零动态内存，但必须知道 `max_size`/`max_count` |
| `FT_POINTER`  | 生成指针，运行时用 `malloc` 动态分配。长度不定的字段用这个                   |
| `FT_CALLBACK` | 生成回调函数指针，编解码时由你的代码流式处理，完全不缓存                         |

Flipper 里 `name`/`data`/`path` 都是长度不定的字符串或大块数据，所以用 `FT_POINTER`，避免为最坏情况预留巨大静态数组。

#### `max_size:N`

字符串/bytes 字段的**最大字节数**。

- 对 `FT_STATIC` 字段：直接决定生成的数组大小 `pb_byte_t data[N]`。
- 对 `FT_POINTER` 字段（像这里的 `File.data`）：nanopb 其实**不强制**它了（注释第 21 行专门说明了这点）——所以这里的 512 只是给人看的约定，server/client 双方要自己遵守，nanopb 生成的代码不会拦你。

#### `max_length:N`

`max_size` 的**字符串专用变体**：表示最大字符数（不含结尾 `\0`），nanopb 会自动 +1 分配 `char[N+1]` 给结尾空字符。语义上比 `max_size` 更适合 C 字符串。

#### `max_count:N`

`repeated` 字段的**最大元素个数**。

```
PB_Storage.ListResponse.file   max_count:8
```

`file` 是个 repeated 字段，一次 List 响应最多返回 8 个 file。对 `FT_STATIC` 的 repeated 字段，这决定生成数组的长度。

#### 其他

- `int_size:IS_8 / IS_16 / IS_32 / IS_64` —— 强制整数字段用更小的 C 类型省内存（proto 里的 int32 默认占 4 字节，可压成 int8）。
- `fixed_length:true` —— bytes 字段固定长度，不再存实际长度。
- `proto3_singular_msgs:true`、`no_unions`、`packed_struct` 等生成细节控制。
- `long_names:false` —— 缩短生成的枚举/常量名。

### 使用

构建时 nanopb 的生成器（`nanopb_generator`）会自动寻找和 `.proto` **同名**的 `.options` 文件，把两者合并后生成 `.pb.c`/`.pb.h`。也就是 `storage.proto` + `storage.options` → `storage.pb.c`。所以：

- 选择器里的名字必须和 [storage.proto](vscode-webview://1gi2okrvrf4hqis68f92gthsi0phd2cfu8u2vek8cg4jkhhbbhmp/assets/protobuf/storage.proto) 里的 message/field 名严格对应。
- 改了 options 需要重新跑 protobuf 生成（Flipper 里通常通过构建脚本触发）。
