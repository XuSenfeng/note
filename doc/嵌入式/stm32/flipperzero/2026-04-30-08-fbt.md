# fbt

fbt是用于flipperzero构建的工具, 是基于https://scons.org/创建的

- **FAM** = **F**lipper **A**pplication **M**anifest,是 Flipper 团队自创的术语;
- 文件名固定为 `application.fam`,语法是**伪 Python**(其实就是直接执行的 Python 代码,但只允许调 `App()` 这个函数)
- 它依赖 scripts/fbt/appmanifest.py 来解析——这是 Flipper 自己写的 ~600 行 Python

使用这个的原因是因为**应用类型多样化**:SERVICE 自动启动、SETTINGS 进设置菜单、EXTERNAL 编 .fap、STARTUP 钩子……这些"角色"必须映射到 Flipper 固件的运行时数据结构里

 `build/f7-updater-D/applications/applications.c` 里那几个 `FLIPPER_SERVICES[]` / `FLIPPER_SYSTEM_APPS[]` 数组吗?**那就是 FBT 解析 `.fam` 后自动生成的 C 代码**);

## APP分类

| 类型               | 启动者            | 启动时机               | 用户感知          | 物理位置                | 占常驻内存   |
| ---------------- | -------------- | ------------------ | ------------- | ------------------- | ------- |
| **SERVICE**      | `flipper_init` | 开机                 | 看不到           | 固件 Flash            | ✅       |
| **STARTUP**      | `flipper_init` | 开机一次               | 看不到           | 固件 Flash            | ❌(只是函数) |
| **APP**          | `loader`       | 用户点主菜单             | 主菜单图标         | 固件 Flash            | ❌       |
| **SYSTEM**       | 其他代码/RPC/CLI   | 被调用                | 看不到入口         | 固件 Flash            | ❌       |
| **SETTINGS**     | `loader`       | 用户点设置菜单            | 设置列表          | 固件 Flash            | ❌       |
| **DEBUG**        | `loader`       | 用户点调试菜单            | Debug 菜单(需开启) | 固件 Flash            | ❌       |
| **ARCHIVE**      | 主菜单            | 长按 OK / Archive 入口 | 文件管理器         | 固件 Flash            | ❌       |
| **EXTERNAL**     | `loader` 动态加载  | 用户点 Apps 子菜单       | Apps 分类       | **SD 卡 .fap**       | ❌       |
| **MENUEXTERNAL** | `loader` 动态加载  | 用户点主菜单             | 主菜单图标         | **SD 卡 .fap**       | ❌       |
| **PLUGIN**       | 宿主 app         | 宿主调用               | 看不到           | SD 卡 .fal / 嵌入 .fap | ❌       |
| **METAPACKAGE**  | 无              | 无                  | 看不到           | 无(只是声明)             |         |

```
                       系统启动
                          │
        ┌─────────────────┴───────────────┐
        ▼                                 ▼
   FLIPPER_SERVICES[]              FLIPPER_ON_SYSTEM_START[]
   (后台线程,常驻)                 (启动钩子,跑完就走)
        │
        ├─ gui_srv (画屏幕)
        ├─ input_srv (按键)
        ├─ storage_srv (SD 卡)
        ├─ loader_srv ◄────── 它是所有用户 App 的启动者!
        ├─ bt_srv
        └─ ...

                       loader_srv 调度
        ┌──────────────┬──────────────┬──────────────┐
        ▼              ▼              ▼              ▼
   FLIPPER_APPS[]  FLIPPER_         FLIPPER_       SD 卡上
   (主菜单)       SETTINGS_APPS[]   DEBUG_APPS[]   /apps/*/*.fap
   subghz          notification     uart_echo     社区 App
   nfc             bt_settings      unit_tests    .fap 动态加载
   ir              ...              ...           (EXTERNAL/MENUEXTERNAL)
   ...
                                                         │
                                                         ▼
                                                    /apps_data/
                                                     插件 .fal
                                                    (PLUGIN)
```

### SERVICE 系统后台服务

**典型**:`gui`、`input`、`storage`、`bt`、`notification`、`loader`、`cli`、`rpc`、`power`
**启动方式**:**系统启动时由 `flipper_init()` 自动创建独立 FreeRTOS 线程并运行**,永不退出

- 占常驻内存,数量受严格控制(整个固件一二十个);
- 通过 `furi_record_open(RECORD_XXX)` 给其他代码提供 API;
- `order` 字段决定启动顺序(storage 必须比依赖它的早起来)。

### STARTUP钩子函数

**典型**:`bt_start`、`updater_start`、`storage` 的初始化钩子
**启动方式**:**只是一个函数指针**,在 `flipper_init()` 启动 SERVICE 之后被**依次调用一次**,跑完就返回

**不创建线程,只是个 callback**。常用来做"开机自检"、"读取上次 RTC 标志做对应处理"(updater_start 检查 PostUpdate 状态就是干这个的)。`order` 决定钩子调用顺序。

### APP 主菜单程序

**典型**:`subghz`、`nfc`、`infrared`、`ibutton`、`badusb`、`u2f`、`gpio`

**启动方式**:用户在**主菜单**点击图标 → 由 `loader` SERVICE 调用 `furi_thread_alloc_ex()` 创建新线程 → 跑 `entry_point` 入口函数。退出后线程销毁,**不占常驻内存**。

- 任意时刻只能跑一个 APP(因为主菜单只允许"进入一个 app");
- 退出回主菜单时,app 完全析构;
- 编译时静态链接到固件 Flash 里。

### SYSTEM后台App

**典型**:`updater_app`、`storage_move_to_sd`、`u2f_init`、`storage_factory_reset`

**启动方式**:**和 APP 一样的线程创建机制,但不出现在任何菜单里**。需要其他代码主动调用 `loader_start("appid", args)` 才能跑起来。

有 UI 能力(可以画界面),但用户**无法从菜单进入**——它只是个"系统功能模块"

### SETTINGS设置菜单项

**典型**:`notification_settings`、`storage_settings`、`bt_settings_app`、`power_settings_app`、`system_settings`
**启动方式**:和 APP 一样的机制(新线程),但**注册到独立的"设置 app 列表"** `FLIPPER_SETTINGS_APPS[]`

**用户接触方式**:**主菜单 → Settings → 选项列表**。点进去就启动对应 SETTINGS app,退出回到设置列表。

**特点**:本质上和 APP 没区别(都是用户主动启动 + 独立线程),只是**所在菜单不同**——APP 在主菜单,SETTINGS 在设置菜单。

### DEBUG调试菜单 App

**典型**:`uart_echo`、`unit_tests`、`display_test`、`led`、`vibro`、`battery_test`

**启动方式**:同 APP,但**只有打开调试模式(Settings → System → Debug = ON)才会出现在菜单里**

`is_default_deployable` 在 appmanifest.py:102 里被设为 False —— 默认不会编进对外发布固件。

### ARCHIVE文件管理器(全系统唯一)

同 APP,但身份特殊:**整个固件只能有一个 ARCHIVE**(枚举上的硬约束)。

**用户接触方式**:**长按主菜单的 OK / 主菜单的 "Archive" 入口**进入,可以浏览 SD 卡上所有 .nfc/.sub/.ir 等文件,直接打开。

**特点**:全局浏览器/调度器,**所有其他 app 都可能向它注册"我能打开 .xxx 文件"**。Flipper 把它从 APP 类型中单独抽出,就是因为它的"唯一全局入口"地位太特殊。

### EXTERNAL .fap 外部应用(SD 卡 App)

**典型**:`example_adc`、社区开发的几百个 app

**启动方式**:**完全不同于上面所有类型** —— 它**不编进固件**,而是单独编译成 `.fap` 文件(ELF 格式 + Flipper 自定义头),**放在 SD 卡上**。运行时由 `loader` 通过自定义 ELF 加载器**动态读 .fap → 解析重定位 → 加载到 RAM → 调用入口**。

**用户接触方式**:**主菜单 → Apps → 分类 → 选择**。SD 卡 `/apps/<fap_category>/<appid>.fap`。

**特点**(这是 Flipper 生态最特殊的设计):

- **不需要重新刷固件就能装 app** —— 这是社区生态的核心;
- 退出后从 RAM 卸载,**不占常驻内存**;
- **ufbt 开发的目标产物**

### MENUEXTERNAL主菜单上的 .fap

**典型**:替代某些内置 APP 的"可拆卸"应用(让某些功能可以从固件中剥离,改放 SD 卡)。

**启动方式**:同 EXTERNAL(.fap 动态加载),但**直接出现在主菜单**而不是 Apps 子菜单。

**用途**:让原本必须编进固件的"主菜单 APP"变成可独立分发的 .fap,**减小固件体积**。

---

### PLUGIN — 插件(给宿主 App 用)

**典型**:NFC 协议解析器、SubGhz 调制解码器、Infrared 厂商解码器

**启动方式**:**它不"启动"** —— PLUGIN 是给**别的 app 加载的扩展**,通过 `plugin_manager_load` 接口。可以两种形式:

- **`fal_embedded=False`**:编成独立 `.fal` 文件放 SD 卡(像 .fap 但是给 app 加载,不是 loader);
- **`fal_embedded=True`**(appmanifest.py:83):嵌入到宿主 .fap 里作为资源,启动时解出来。

```python
App(
    appid="nfc_protocol_emv_plugin",
    apptype=FlipperAppType.PLUGIN,
    entry_point="nfc_emv_plugin_ep",
    requires=["nfc"],
)
```

**用户接触方式**:**间接** —— 比如刷一张银行卡,NFC app 会自动找到并加载 EMV plugin 来解析,用户感知不到 plugin 的存在。

**特点**:**插件是给开发者扩展系统能力的、不是给用户启动的**。让"协议支持"等可扩展能力做到模块化。


### METAPACKAGE元包(纯构建概念)

**典型**:`basic_services`、`main_apps`、`debug_apps`

**启动方式**:**完全没有代码,纯粹的依赖声明容器**。

```python
App(
    appid="main_apps",
    apptype=FlipperAppType.METAPACKAGE,
    requires=["subghz", "nfc", "ir", "ibutton", "badusb", "u2f", "gpio"],
)
```

**用户接触方式**:**完全不可见**。

**特点**:类似 Linux 的 "meta package" 概念。构建配置里写 `--extra-int-apps=main_apps`,FBT 就会自动把它依赖的一串 app 都拉进来。是**构建配置的简写**。


## 环境变量

这个文件实际是一个脚本文件, 默认的环境变量不会影响全局的变量, 但是可以使用命令创建一个包含环境变量的shell

>   -   Windows 下，直接运行 [fbtenv.cmd](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html)。
>   -   Linux 和 MacOS 下，在新 shell 里运行 `source scripts/toolchain/fbtenv.sh`。
>   -   你也可以在 shell 里输入 `. `./fbt -s env` `（注意开头的点）。

如果你的系统不支持预编译工具链，或你想用自定义依赖版本，可以设置 `FBT_NOENV=1`。此时 `fbt` 会跳过工具链和环境配置，要求所有工具都在系统 PATH 中可用。

如果设置了 `FBT_TOOLCHAIN_PATH` 变量，`fbt` 会将工具链解压到该目录。默认会下载到仓库根目录下的 `toolchain` 子目录。

如需为 `fbt` 及工具链管理脚本启用额外调试输出，可以设置 `FBT_VERBOSE=1`

fbt` 启动时总会执行 `git submodule update --init`，除非你在环境变量中设置 `FBT_NO_SYNC=1

>   -   Windows 下，在运行 `fbt` 的 shell 里设置 `set "FBT_NO_SYNC=1"`
>   -   *nix 下，使用 `$ FBT_NO_SYNC=1 ./fbt ...`

## 构建

`fbt` 会在 `build` 目录下为 updater 和固件分别创建子目录，名称取决于优化设置（`COMPACT` 和 `DEBUG` 选项）。为方便 IDE 集成，最新构建变体的目录总是链接为 `build/latest`。此外，`compile_commands.json` 也会生成在该文件夹（用于 IDE 代码补全）。

只有*固件构建目标*会更新 `build/latest` 软链接和编译数据库。运行其他任务（如固件烧录或为不同配置/硬件目标构建更新包）不会更新 `build/latest`

## SCons使用

需要创建文件`SConstruct`用于编译

SCons 支持的编译类型有：

+   Program： 编译成可执行程序（在 Windows 平台上即是 exe 文件），这是最常用的一种编译类型。
+   Object： 只编译成目标文件。使用这种类型，编译结束后，只会产生目标文件。在 POSIX 系统中，目标文件以 .o 结尾，在 Windows 平台上以 .OBJ 结尾。
+   Library： 编译成库文件。SCons 默认编译的库是指静态链接库。
+   StaticLibrary： 显示的编译成静态链接库，与上面的 Library 效果一样。
+   SharedLibrary： 在 POSIX 系统上编译动态链接库，在 Windows 平台上编译 DLL。

### 初级使用

```bash
Program("hello.c")
```

这个C文件就是使用的代码文件, 在这里，没有指定生成的可执行程序的名字，SCons 会把源代码 文件名字的后缀去掉，用来作为可执行文件的名字

### 各种参数

+ 可执行文件的名字

```python
Program('myhello', 'helloscons.c')
```

+   清理

不需要像 Makefile 那样指定清理的动作，就可以执行清理任务。在 SCons 中，执行清理任务由参数 -c 指定

```bash
scons -c
```

+   简化输出

```bash
scons -Q
```

+   复杂编译

如果你的项目由多个源文件组成，而且你想指定一些编译的宏定义，以及显式的指定使用某些库

```python
Program('helloscons2', ['helloscons2.c', 'file1.c', 'file2.c'], 
       LIBS = 'm', 
       LIBPATH = ['/usr/lib', '/usr/local/lib'], 
       CCFLAGS = '-DHELLOSCONS')
```

如果你的源程序代码文件很多，有十几个甚至上百个，那不要一个个的将他们都列出来，你可以使用 glob(’*.c’) 来代替源代码列表

>   +   `LIBS`： 显示的指明要在链接过程中使用的库，如果有多个库，应该把它们放在一个列表里面。这个例子里，我们使用一个称为 m 的库。
>   +   `LIBPATH`： 链接库的搜索路径，多个搜索路径放在一个列表中。这个例子里，库的搜索路径是 /usr/lib 和 /usr/local/lib。
>   +   `CCFLAGS`： 编译选项，可以指定需要的任意编译选项，如果有多个选项，应该放在一个列表中。这个例子里，编译选项是通过 -D 这个 gcc 的选项定义了一个宏 HELLOSCONS。
>   +   `CPPPATH`：指定头文件的路径

+   指定环境变量

```bash
env = Environment(CC = 'g++')
env.Program("client", "client.c", LIBS = 'm', CPPPATH = '../include', CCFLAGS = '-std=c++11')
```
