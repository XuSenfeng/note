## 蓝牙方案
+ SOC 单芯片

半导体开发厂商直接把开发的蓝牙协议烧录到使用的芯片里面, 提供一个半开源的协议栈(ESP32)

一般可以直接作为一个MCU进行使用, 进行消费电子, 集成度比较高

+ SOC + MCU
在集成好的蓝油协议基础上, 使用特定的接口, 比如串口, 发送自定义的command来实现想要开发的功能
蓝牙实际是一个外设, 可以用于远程通信, 网上售卖的蓝牙模块

+ 蓝牙host + controlller
最复杂的一种应用, 用于实现不同的蓝牙协议, 把Host和Controller分开, 实现蓝牙音频, 蓝牙电话等, 两者之间通过Transport协议进行连接, H2基于USB, H4, H5, BCSP基于UART

## Linux蓝牙

BlueZ是Linux官方蓝牙协议栈，由蓝牙 SIG（Special Interest Group）认证，支持蓝牙1.1至5.4全版本，涵盖经典蓝牙（BR/EDR）和低功耗蓝牙（BLE）。其核心功能包括：

- **内核模块**：提供驱动框架、HCI（主机控制器接口）协议、L2CAP（逻辑链路控制与适配协议）等底层支持。
- **用户空间工具**：如`bluetoothctl`（设备管理）、`hcitool`（HCI命令）、`btmon`（日志分析）等。
- **服务守护进程**：`bluetoothd`负责设备发现、配对、连接管理等高层逻辑。

Linux蓝牙架构严格遵循“内核-用户空间分离”原则：

- **内核空间**：
    - **驱动层**：直接与硬件交互（如USB蓝牙适配器、UART蓝牙模块），实现HCI协议。
    - **核心协议层**：L2CAP、SDP（服务发现协议）、RFCOMM（串口仿真）等，为上层提供标准化接口。

> HCI: 蓝牙主机和控制器之间的接口, 负责管理蓝牙的硬件资源
> L2CAP: 为上层协议提供数据封装以及分段服务, 用于逻辑链路的建立以及配置
> SDP: 查询以及发现蓝牙网络里面的可以使用的服务
> RFCOMM: 基于L2CAP实现的可以用于串口的模拟
> OBEX: 用于文件夹的传输以及实现文件的共享

- **用户空间**：
    - **服务层**：`bluetoothd`通过D-Bus暴露API，供应用（如桌面环境、IoT设备）调用。
    - **工具层**：命令行工具（如`hcitool`）和图形界面工具（如GNOME Bluetooth），用于调试和用户交互。

蓝牙协议栈自上而下可分为：

- **应用层**：具体应用（如音频播放、文件传输）。
- **会话层**：SDP（服务发现）、AVCTP（音频/视频控制）等。
- **传输层**：L2CAP（多路复用逻辑链路）、RFCOMM（模拟串口）。
- **主机控制器接口（HCI）**：连接主机（Linux系统）与控制器（蓝牙芯片）的标准接口，定义了命令、事件、数据的格式。
- **控制器层**：蓝牙硬件（射频、基带、链路管理），负责物理层和链路层实现。

其中，**HCI**是驱动开发的核心关注点：内核驱动需将Linux的HCI命令转换为硬件可执行的指令（如通过USB、UART发送给蓝牙芯片）

### 控制逻辑

BlueZ在用户空间和内核空间之间采用socket API进行交互。用户空间程序通过标准的socket接口调用内核中的蓝牙服务。交互流程通常包括：

1. socket创建 : 在用户空间创建socket并绑定到特定的蓝牙协议。
2. 服务发现 : 通过socket API查询和发现服务。
3. 连接管理 : 用户空间程序通过socket API发起连接和断开连接。
4. 数据交换 : 通过socket API发送和接收数据。
5. 这种机制实现了灵活和高效的数据处理能力，同时也保证了内核的稳定性和安全性。

BlueZ提供一系列内核API供内核模块和驱动程序使用，主要的API有：

+ HCI API : 提供对蓝牙控制器的访问和管理。作用是在「蓝牙主机（Host，即运行协议栈的 CPU / 操作系统）」和「蓝牙控制器（Controller，即蓝牙射频芯片 / 固件）」之间做抽象隔离
+ SCO API : 管理声音传输相关的SCO链接。
+ L2CAP API : 管理L2CAP连接和相关的数据传输。
+ RFCOMM API : 创建和管理RFCOMM通道。
+ ATT API : 用于管理蓝牙低功耗设备的ATT/GATT层。

通过这些内核API，开发者可以实现定制化的蓝牙驱动和协议栈扩展。

```bash
┌─────────────────────────────┐ 
│    应用层（APP/业务逻辑）      │ ← 绝大多数开发者在这里 
├─────────────────────────────┤ 
│    系统服务 & 上层协议层       │ ← D-Bus、GATT、RFCOMM、L2CAP ├─────────────────────────────┤ 
│    HCI 核心 & 传输驱动层      │ ← HCI 所在层，底层开发核心 
└─────────────────────────────┘ 
蓝牙硬件控制器
```
- BlueZ 5.x 之后官方推荐的标准开发方式是 **D-Bus API**，所有蓝牙能力（扫描、连接、配对、GATT 读写、音频连接等）都通过 D-Bus 接口提供
- 上层语言的蓝牙库（如 Python 的 `bluepy`、`bleak`，Qt 的 `QtBluetooth`）底层都是对接 D-Bus 或系统蓝牙服务
