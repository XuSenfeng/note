# 定时任务

OpenWrt默认使用busybox工具箱软件自带的cron系统功能

可以使用命令配置时钟

```bash
crontab -e 
```

> 这个命令可以打开`/etc/crontabs/root`文件, 因为OpenWrt默认的登录用户名是root，某个用户的定时任务配置文件，就放在以它的用户名命名的文件

可以使用的命令

```bash
crontab -l
```

查看现在已经有的定时任务, 每一个任务的配置占一行, 不同的任务使用一个换行结束

```bash
* * * * * command to execute
- - - - -
| | | | |
| | | | ----- Day of week (0 - 6) (Sunday =0)
| | | ------- Month (1 - 12)
| | --------- Day (1 - 31)
| ----------- Hour (0 - 23)
------------- Minute (0 - 59)
```

| 分钟 0-59 | 小时 0-23 | 月中的第几天 1-31 | 月份 1-12 | 星期中的第几天 0-6 | 执行效果                                                     |
| :-------: | :-------: | :---------------: | :-------: | -----------------: | :----------------------------------------------------------- |
|    */5    |     *     |         *         |     *     |                  * | 每5分钟执行一次                                              |
|    12     |    */3    |         *         |     *     |                  * | 每过3个小时后的第12分钟执行一次                              |
|    57     |    11     |        15         |  1,6,12   |                  * | 在1、6、12月中的15豪，当天的11点57分各执行一次               |
|    25     |     6     |         *         |     *     |                1-5 | 工作日期间（周1到周5），每天早上6点25分执行一次              |
|     0     |     0     |      4,12,26      |     *     |                  * | 每月的第4、12、26日，晚上12点执行一次                        |
|   5,10    |   9,14    |        10         |     *     |                0,4 | 每周日、周四的早上9:05, 早上9:10, 下午2:05 以及下午2:10各执行一次 |

## 执行日志

```bash
logread -e cron
```

不是所有信息都会被记录到日志里的，要修改记录的信息等级，可以用“cronloglevel”属性，

## 应用

### 定时重启

可以用于定期的内存的回收, 解决性能的下降

但是实际重启的时候可能会出现一些奇怪的问题, 这是因为每一次重启的时候时间还没有被设置, 当前系统的时间使用“sysfixtime”程序来设置；设定依据是在/etc目录下，使用**最近被修改的文件的时间**。 它可能是一个状态文件，或者配置文件，可能在cron定时任务需要重启系统前30秒刚做了修改。 所以，如果在重启系统的时候，将系统时间设置为这个文件的修改时间的前若干秒钟的话， 当cron软件开始计时的时候，马上又会发现cron设置的重启时间又要到了，接着**又开始重启了**

因为启动流程的最后ntpd的时间服务启动了，这个软件得需要一阵子才能获取并且设置正确的时间，所以这段时间内，cron定时重启任务又被触发重启了

一种解决cron定时重启任务的怪圈的办法是在重启前，**推迟**一会儿重启，同时主动用touch命令去**修改/etc下的任意**文件

````bash
# Reboot at 4:30am every day
# Note: To avoid infinite reboot loop, wait 70 seconds
# and touch a file in /etc so clock will be set
# properly to 4:31 on reboot before cron starts.
30 4 * * * sleep 70 && touch /etc/banner && reboot
````

> 1. `30 4 * * *`：每天 **4:30** 触发任务
> 2. `sleep 70`：**等待 70 秒**（给 NTP 时间同步服务足够的时间校准正确时间）
> 3. `touch /etc/banner`：修改 `/etc/banner` 文件的时间戳为**当前正确时间**
> 4. `reboot`：执行路由器重启

在一些平台上“shutdown”命令是没有的；它仅仅是停止CPU的运行，而不会重启设备。 这通常是因为这样的设备没有去执行断电的电路。

即便这样，也还是可以使用一种更为灵活的方法，通过使用[watchcat](https://openwrt.org/packages/pkgdata/watchcat)软件包来使用 。

```bash
opkg update
opkg install watchcat  # 安装核心功能包
opkg install luci-app-watchcat  # 安装Web管理界面（推荐）
opkg install luci-i18n-watchcat-zh-cn  # 安装中文语言包
```

> 在 OpenWRT 系统中安装网络监控守护进程（ping-watchdog），核心作用是自动检测网络连接状态并执行预设恢复动作，保障路由器稳定运行
>
> Watchcat 是一个轻量级网络监控工具，通过定期 ping 指定主机来判断网络连通性
>
> | 模式                  | 触发条件                   | 执行动作                      | 典型应用场景                            |
> | --------------------- | -------------------------- | ----------------------------- | --------------------------------------- |
> | **Ping Reboot**       | 连续 ping 失败达到设定次数 | 重启整个路由器                | 解决 ISP 断网、路由器死机等严重网络问题 |
> | **Restart Interface** | 特定接口上 ping 失败       | 重启指定网络接口（如 WAN 口） | PPPoE 连接异常、VPN 断开等局部网络问题  |
> | **Periodic Reboot**   | 达到设定时间周期           | 按计划重启设备                | 释放内存、解决长期运行导致的性能下降    |
>
> **Ping 目标**：选择可靠主机（如 8.8.8.8、1.1.1.1 等公共 DNS）
>
> **检测间隔**：设置 ping 检测频率（建议 5-10 分钟）
>
> **失败阈值**：连续 ping 失败多少次后触发动作
>
> **恢复动作**：选择重启设备、重启接口或仅记录日志

### 闹钟

已使用openWRT实现其他设备的定时开启

```bash
59 05 * * 1 /usr/bin/wol -h 192.168.1.255 xx:xx:xx:xx:xx:xx # Mo

```

**每周一早上 5 点 59 分，路由器自动唤醒指定 MAC 地址的电脑 / 设备**

```bash
opkg update
opkg install wol
```

被唤醒的设备：**BIOS 开启 Wake-on-LAN** + **网卡开启网络唤醒**
