## 下载固件

**树莓派 4B 官方固件下载**：https://downloads.openwrt.org/releases/23.05.5/targets/bcm27xx/bcm2711/

下载文件：`openwrt-23.05.5-bcm27xx-bcm2711-rpi-4-squashfs-factory.img.gz`

+   `openwrt-23.05.5`：OpenWRT 版本号（23.05 系列第 5 个稳定版）
+   `bcm27xx-bcm2711`：目标芯片平台（树莓派 4B 使用的 Broadcom BCM2711 芯片）
+   `rpi-4`：具体设备型号（Raspberry Pi 4）
+   `squashfs`：根文件系统格式
+   `factory`：刷写场景（第一次烧录）
+   `.img`：原始磁盘镜像格式
+   `.gz`：gzip 压缩格式（烧录工具会自动解压）

`squashfs-factory`是 OpenWRT 固件文件名中**两个独立关键标识的组合**，分别定义了固件的**根文件系统格式**和**刷写场景**

SquashFS 是一种**高压缩率的只读 Linux 文件系统**，是 OpenWRT 官方推荐的标准格式，专为嵌入式设备有限的存储空间设计。

+   工作原理

OpenWRT 采用 **"只读 SquashFS + 可写 OverlayFS"** 的双分区架构：

>    **SquashFS 分区**：存放系统核心文件（内核、驱动、基础软件），**永远只读**，任何修改都不会改变这个分区**OverlayFS 分区**：存放用户的所有配置和安装的软件，**可写**，位于 SD 卡的剩余空间

+   固件模式

`factory`表示这是**工厂固件**，专门用于**设备还在运行原厂系统时的第一次刷写**。

对比：factory vs sysupgrade（另一种常见固件类型）

| 对比维度 | factory 固件                                               | sysupgrade 固件                       |
| -------- | ---------------------------------------------------------- | ------------------------------------- |
| 刷写场景 | 第一次刷写（原厂系统→OpenWRT）                             | 系统升级（OpenWRT→新版 OpenWRT）      |
| 包含内容 | 完整的系统镜像（包含分区表、bootloader、内核、根文件系统） | 仅包含内核和根文件系统                |
| 刷写方式 | 用 SD 卡烧录工具（如 Raspberry Pi Imager）直接写入 SD 卡   | 在 OpenWRT 的 LuCI 界面或命令行中升级 |
| 数据保留 | 会清除所有数据                                             | 默认保留用户配置                      |

1. 用**上下键**选中最后一行的 **Free space**（空闲空间）。
2. 按**左右键**选择 **[New]** → 回车。
3. 直接回车（使用**全部剩余空间**）。
4. 选择 **primary** → 回车。
5. 按左右键选择 **[Type]** → 输入 `83` (Linux 文件系统) → 回车。
6. 按左右键选择 **[Write]** → 输入 `yes` → 回车（**保存分区表**）。
7. 选择 **[Quit]** 退出。

```bash
fdisk -l /dev/mmcblk0 # 查看创建的分区的名字
mkfs.ext4 /dev/mmcblk0p3 # 格式化
# 临时挂载
mkdir -p /mnt/sda3
mount /dev/mmcblk0p3 /mnt/sda3
# 迁移数据(可能失败/文件夹为空)
cp -a /overlay/* /mnt/sda3/
# 自动挂载
# 清除旧配置
uci -q delete fstab.overlay
# 设置新挂载
uci set fstab.overlay="mount"
uci set fstab.overlay.device="/dev/mmcblk0p3"
uci set fstab.overlay.target="/overlay"
uci set fstab.overlay.fstype="ext4"
uci set fstab.overlay.enabled="1"
# 保存
uci commit fstab

reboot
```

### 原理

OpenWrt 为了**防崩溃、防闪存损坏、好升级**，设计成了**双层分离架构**：

1. 上层：只读系统分区（固件本体）

对应你的：`/dev/mmcblk0p2`

- 格式：压缩只读文件系统 **SquashFS**

- 特点：

    - 内核、驱动、基础命令、网络核心配置 全在这里
    - **完全不能写入、不能修改**
    - 体积很小（官方固件就几十 MB）

    

- 作用：保证系统本体永远纯净、不坏、不乱改

2. 下层：可读写数据分区（overlay）

挂载目录：`/overlay`

- 唯一**能写入、能保存数据**的地方
- 所有改动全部存在这里：
    - 安装的插件 / 软件
    - 你改的网络配置、密码、后台设置
    - 缓存、日志、docker、自定义脚本

系统原理：`OverlayFS 联合挂载`

把「只读系统 p2」+「可写 overlay」拼在一起，拼成你看到的完整 `/` 根目录。

初始的SD卡

```bash
/dev/mmcblk0   整张SD卡
 ├─ p1  boot    启动分区（FAT32，引导树莓派开机，不动）
 ├─ p2  root    只读系统固件（不可写、不能扩容）
 └─ p3  我们新建的空分区（整块剩余大容量空间，ext4）
```

SD 卡除了 p1/p2，剩下几十 G 都是空白未分配，

我们新建一个独立 Linux 分区，专门拿来当**大容量 overlay**。OpenWrt 的 overlay 必须用 Linux 标准文件系统 `ext4`，FAT32/NTFS 权限不达标，不能当做系统读写目录。

+ OpenWrt 开机流程：

1. 先加载只读系统 → 读取`fstab`挂载表 → 自动挂载 `/overlay`

原本：系统自动用「p2 里一小块极小空间」挂载为 `/overlay`

改完之后：强制让系统**开机自动把大容量分区 `/dev/mmcblk0p3` 挂载到 /overlay**
