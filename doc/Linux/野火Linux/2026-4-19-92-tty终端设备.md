# tty终端设备

在Linux 系统中，终端是一种字符型设备，它有多种类型，通常使用tty来简称各种类型的终端设备(Teletype) , ，Teletype 是最早出现的一种终端设备，很像电传打字机，是由Teletype 公司生产的

**“终端设备” 本质就是：一套用来做「输入 + 输出」的接口**。

程序不关心你是人、是硬件还是别的程序，它只认：

**能往里写、能往外读、像串口 / 键盘显示器那样交互的设备 = 终端设备**

## 常见类型

### 串行端口终端（/dev/ttySn）

使用计算机串行端口连接的终端设备。计算机把每个 串行端口都看作是一个字符设备

在命令行上把标准输出重定向到端口对应的设备文件名上就可以通过该端口发送数据，例 如，在命令行提示符下键入：echo test > /dev/ttyS1会把单词“test”发送到连接在ttyS1端口的 设备上

### 伪终端（/dev/pty/）

伪终端pty（Pseudo Terminal）是成对的逻辑终端设备，并存在成对的设备文件

+ 一头叫 **主设备**：`/dev/ptyp3`
+ 一头叫 **从设备**：`/dev/ttyp3`

它们与实际物理设备并不直接相关。如果一个程序把 ttyp3 看作是一个串行端口设备，则它对该端口的读/写操作会反映在该逻辑终端设备对应的ttyp3上，而ttyp3则是另一个程序 用于读写操作的逻辑设备。这样，两个程序就可以通过这种逻辑设备进行互相交流，使用 ttyp3 的程序会认为自己正在与一个串行端口进行通信

+ 程序 A 往 ttyp3 写数据 → 数据**自动跑到对面的 ptyp3**
+ 程序 A 从 ttyp3 读数据 → 实际读的是 ptyp3 发过来的数据

现在 Linux 用的是**新式 PTY**：

- 主设备：`/dev/ptmx`
- 从设备：`/dev/pts/0`、`/dev/pts/1`……

- **主 PTY（master pty，如 /dev/ptyp3）**：是**内核提供的虚拟 “读写端口”**，相当于**管道的一头**。
- **Shell（bash、sh）**：是**跑在从 PTY 上的应用程序**，负责解析命令、执行程序。

它们的关系是：**终端模拟器持有主 PTY，Shell 绑定在从 PTY 上**。

1. 你敲 `ls`
2. 终端模拟器 → 写给 **主 PTY (ptyp3)**
3. 内核自动转发 → **从 PTY (ttyp3)**
4. *Shell(bash)** 从 ttyp3 读到 `ls`
5. Shell 执行 ls，把结果写到 ttyp3
6. 内核转发 → ptyp3
7. 终端模拟器显示出来

### 控制终端（/dev/tty）

如果当前进程有控制终端（Controlling Terminal）的话，那么/dev/tty就是当前进程的控制终端 的设备特殊文件

可以使用命令“ps –ax”来查看进程与哪个控制终端相连，使用命令“tty”可以 查看它具体对应哪个实际终端设备

不是一种硬件，而是**一个 “归属关系”**：

一个程序运行时，系统给它绑定一个终端，作为它默认的输入 / 输出、发信号的通道，这个终端就叫它的**控制终端**；是**进程和某个终端的绑定关系**（谁是这个进程的交互入口）

可以是：控制台终端（tty2）、伪终端（/dev/pts/0）、串口终端，**任何终端都能当控制终端**。

### 控制台终端（/dev/ttyn，/dev/console）

+ 是**操作系统最原始、最核心的交互终端**，和硬件强相关；
+ 用来显示**开机日志、内核崩溃信息、系统级报错**，普通终端炸了它还在；

1. **数量固定**：一般就 6 个（tty1~tty6）；
2. **系统级**：内核直接管理，不依赖图形界面；
3. **优先级最高**：图形界面崩了，控制台还能用。

在 UNIX 系统中，计算机显示器通常被称为控制台终端（console）。 它仿真了类型为 Linux 的一种终端（TERM=Linux）， 并且有一些设备特殊文件与之相关联 ： tty0、tty1、tty2等。当用户在控制台上登录时，使用的是tty1。使用`Alt+[F1～F6]`组合键时，我们就可以切换到tty2、tty3等 上面去。tty1～tty6 等称为虚拟终端，而tty0则是当前所使用虚拟终端的一个别名

在Linux 中，可以在系统启动命令行里指定当前的输出终端, 是**Linux 内核的启动参数**, 告诉内核：系统刚开机、还没进系统界面时，所有启动日志、报错、内核打印信息，输出到哪个「终端设备」上。

```txt
console=设备名,参数选项
```

| 设备名    | 含义                              | 适用场景                                     |
| --------- | --------------------------------- | -------------------------------------------- |
| `tty1`    | 系统控制台（显示器输出）          | 接了屏幕的 Jetson Nano，开机日志在屏幕上显示 |
| `ttyS0`   | 物理串口 0（UART 串口）           | Jetson Nano 调试串口、无显示器远程调试       |
| `ttyAMA0` | ARM 平台通用串口（和 ttyS0 类似） | 树莓派 / Jetson 部分版本                     |
| `ttyUSB0` | USB 转串口设备                    | USB 串口模块连接开发板                       |

参数的使用一般是`波特率,校验位,数据位,停止位`比如`115200n81`, 可以同时指定多个, 参数是可选的

## 驱动

![image-20260419210732349](https://picture-01-1316374204.cos.ap-beijing.myqcloud.com/lenovo-picture/202604192107414.png)

**tty 设备发送数 据的流程为**

1. tty核心从一个用户获取将要发送给一 个tty 设备的数据
2. tty核心将数据传递给tty线路规程驱动，接着数据被传递到tty驱动
3. tty驱动将数 据转换为可以发送给硬件的格式

尽管大多数时候tty核心和tty之间的数据传输会经历tty线路规程的转换，但是 tty 驱动与tty 核心之间也可以直接传输数据

`drivers/char/tty_io.c` 定义了 tty 设备通 用的 `file_operations` 结构体并实现了接口函数 `tty_register_driver()`用于注册 tty 设备，它会利用 fs/char_dev.c 提供的接口函数注册字符设备

与具体设备对应的 tty 驱动将实现 tty_driver 结构体 中的成员函数

tty_io.c也提供了tty_register_ldisc()接口函数用于注册线路规程，典型地，例 如`drivers/char/n_tty.c` 文件则针对 N_TTY 线路规程实现了tty_disc 结构体中的成员

![image-20260419211146958](https://picture-01-1316374204.cos.ap-beijing.myqcloud.com/lenovo-picture/202604192111017.png)

1. 字符设备通用层 (fs/char_dev.c)

- **作用**：Linux 所有字符设备的统一入口，提供字符设备的注册、管理和文件系统接口
- **核心操作**：TTY 子系统向这里注册**主设备号为 4**的字符设备，提供通用的`struct file_operations`接口
- **用户态入口**：当你执行`open("/dev/ttyS0")`时，首先进入这一层，然后转发到 TTY 核心层

2. TTY 核心层 (tty_io.c)

**这是整个 TTY 子系统的大脑和调度中心，承上启下，不做任何具体业务逻辑**

- 核心职责
    1. 管理所有 TTY 设备实例（`struct tty_struct`）
    2. 接收底层硬件驱动的注册
    3. 接收行规程的注册
    4. 转发用户态请求到底层，转发硬件数据到上层

| 接口                    | 注册对象            | 作用                   |
| ----------------------- | ------------------- | ---------------------- |
| `tty_register_driver()` | `struct tty_driver` | 注册具体的串口硬件驱动 |
| `tty_register_ldisc()`  | `struct tty_disc`   | 注册终端行规程         |

3. TTY 硬件驱动层 (xxx_tty.c)

- **作用**：直接和物理串口硬件打交道，负责原始字节的收发
- **举例**：16550 串口驱动、USB 转串口驱动、ARM 平台的 PL011 串口驱动
- **核心实现**：每个具体的串口硬件都有一个对应的驱动，实现`struct tty_driver`接口
- **数据交互**：通过中断方式接收硬件数据，通过寄存器操作发送数据到硬件

4. TTY 行规程层 (n_tty.c)

**这是 TTY 子系统最有特色、最容易被误解的一层**

- **默认实现**：`n_tty`（New TTY），是 Linux 默认的行规程
- **核心职责**：实现所有终端交互逻辑，**硬件驱动只收发原始字节，所有终端功能都在这里完成**
- 典型功能
    - 字符回显（echo）：你输入一个字符，屏幕上显示出来
    - 行编辑：退格、删除、光标移动
    - 信号处理：Ctrl+C 产生 SIGINT、Ctrl+Z 产生 SIGTSTP
    - 换行转换：CR/LF 转换（串口默认发送 \r\n）
    - 流量控制：XON/XOFF 软件流控

```bash
用户态调用 write("/dev/ttyS0", buf, len)
    ↓
字符设备层 char_dev.c 转发
    ↓
TTY核心层 tty_io.c 找到对应的tty_struct
    ↓
行规程层 n_tty.c 处理（回显、换行转换、流控检查）
    ↓
TTY核心层 转发到对应的硬件驱动
    ↓
硬件驱动 xxx_tty.c 将数据写入串口寄存器
    ↓
物理串口硬件发送数据

物理串口硬件收到数据，产生中断
    ↓
硬件驱动 xxx_tty.c 读取数据，交给TTY核心层
    ↓
行规程层 n_tty.c 处理（回显、行编辑、信号检测）
    ↓
数据存入 tty_struct 的读缓冲区
    ↓
用户态调用 read() 从缓冲区读取数据
```

## 数据类型

### tty

![image-20260422231239614](https://picture-01-1316374204.cos.ap-beijing.myqcloud.com/lenovo-picture/202604222312637.png)

Linux TTY 子系统的核心数据结构之一。在 Linux 内核中，所有的字符终端设备（例如**串口（ttyS, ttyUSB）、物理控制台（tty0）、伪终端（PTY, dev/pts）等**）都要通过实现并注册这个结构体来与系统内核建立联系

这个结构体主要用于描述一个 TTY **驱动程序**的整体信息（不是单个设备的运行状态，单个设备由 `tty_struct` 描述）。驱动程序的初始化、分配设备号、定义硬件操作方法等都要通过它来实现

```c
struct tty_driver {
    // 用于在运行时快速校验该结构体指针的合法性
	int	magic;		/* magic number for this structure */
	struct kref kref;	/* Reference management */
	struct cdev **cdevs;
	struct module	*owner;
    // 驱动的名称（用于 /proc/tty/drivers 等系统信息呈现），如 "serial"
	const char	*driver_name;
    // 设备节点名称的前缀。比如这里设为 "ttyS"，内核就会在 
    // /dev/ 下生成 ttyS0、ttyS1 等设备
	const char	*name;
    // 节点的起始编号
	int	name_base;	/* offset of printed name */
	int	major;		/* major device number */
	int	minor_start;	/* start of minor device number */
	unsigned int	num;	/* number of devices allocated */
    // 驱动的类型和子类型。例如类型可以是串口 TTY_DRIVER_TYPE_SERIAL 
    // 或伪终端 TTY_DRIVER_TYPE_PTY
	short	type;		/* type of tty driver */
	short	subtype;	/* subtype of tty driver */
    // 初始的终端控制设置（如波特率、流控、字符大小等）。
    // 所有新打开的该类型设备都会默认继承这个 termios 设定
	struct ktermios init_termios; /* Initial termios */
	unsigned long	flags;		/* tty driver flags */
	struct proc_dir_entry *proc_entry; /* /proc fs entry */
	struct tty_driver *other; /* only used for the PTY driver */

	/*
	 * Pointer to the tty data structures
	 */
    // 存放具体正在工作的各个终端的上下文对象
	struct tty_struct **ttys;
    // 存放底层端口状态
	struct tty_port **ports;
    // 存放每个终端各自独立的 termios 配置，覆盖初始化时的默认配置
	struct ktermios **termios;
    // 驱动程序的私有全局数据，留给具体的底层驱动自己挂载需要的私有结构
	void *driver_state;

	/*
	 * Driver methods
	 */
	// 包含了所有“底层硬件如何执行动作”的函数指针
	const struct tty_operations *ops;
	struct list_head tty_drivers;
} __randomize_layout;
```

初始的标准的串口的配置如下

### 操作函数

```c
struct tty_operations {
    // 设备生命周期
    // 按索引找现有 tty
	struct tty_struct * (*lookup)(struct tty_driver *driver,
			struct file *filp, int idx);
    // 把一个新 tty 装进驱动内部表
	int  (*install)(struct tty_driver *driver, struct tty_struct *tty);
    // 把关闭的 tty 从内部表移除
	void (*remove)(struct tty_driver *driver, struct tty_struct *tty);
	int  (*open)(struct tty_struct * tty, struct file * filp);
	void (*close)(struct tty_struct * tty, struct file * filp);
	void (*shutdown)(struct tty_struct *tty);
	void (*cleanup)(struct tty_struct *tty);
    
    // 数据传输
	int  (*write)(struct tty_struct * tty,
		      const unsigned char *buf, int count);
	int  (*put_char)(struct tty_struct *tty, unsigned char ch);
    // 把暂存字符真正刷出去
	void (*flush_chars)(struct tty_struct *tty);
    // 还能写多少
	int  (*write_room)(struct tty_struct *tty);
    // 当前缓冲里还有多少未发数据
	int  (*chars_in_buffer)(struct tty_struct *tty);
	int  (*ioctl)(struct tty_struct *tty,
		    unsigned int cmd, unsigned long arg);
	long (*compat_ioctl)(struct tty_struct *tty,
			     unsigned int cmd, unsigned long arg);
    // 终端参数变化时通知驱动
	void (*set_termios)(struct tty_struct *tty, struct ktermios * old);
	void (*throttle)(struct tty_struct * tty);
	void (*unthrottle)(struct tty_struct * tty);
	void (*stop)(struct tty_struct *tty);
	void (*start)(struct tty_struct *tty);
	void (*hangup)(struct tty_struct *tty);
	int (*break_ctl)(struct tty_struct *tty, int state);
    // 刷新缓冲区并丢弃任何剩下的数据
	void (*flush_buffer)(struct tty_struct *tty);
    // 设置线路规程, 当tty核心改变tty驱动的线路规程时这个函数被调用
	void (*set_ldisc)(struct tty_struct *tty);
	void (*wait_until_sent)(struct tty_struct *tty, int timeout);
    // 发送高优先级控制字符
	void (*send_xchar)(struct tty_struct *tty, char ch);
	int (*tiocmget)(struct tty_struct *tty);
	int (*tiocmset)(struct tty_struct *tty,
			unsigned int set, unsigned int clear);
	int (*resize)(struct tty_struct *tty, struct winsize *ws);

	/* only for abi preservation */
	int (*set_termiox)(struct tty_struct *tty, struct termiox *tnew);
	
    // 获取串口统计信息
	int (*get_icount)(struct tty_struct *tty,
				struct serial_icounter_struct *icount);
	int  (*get_serial)(struct tty_struct *tty, struct serial_struct *p);
	int  (*set_serial)(struct tty_struct *tty, struct serial_struct *p);
	void (*show_fdinfo)(struct tty_struct *tty, struct seq_file *m);
#ifdef CONFIG_CONSOLE_POLL
	int (*poll_init)(struct tty_driver *driver, int line, char *options);
	int (*poll_get_char)(struct tty_driver *driver, int line);
	void (*poll_put_char)(struct tty_driver *driver, int line, char ch);
#endif
	int (*proc_show)(struct seq_file *, void *);
} __randomize_layout;
```

### 注册示例

```c
static int vcc_tty_init(void)
{
	int rv;

	pr_info("VCC: %s\n", version);
	// 分配数据
	vcc_tty_driver = tty_alloc_driver(VCC_MAX_PORTS, VCC_TTY_FLAGS);
	if (IS_ERR(vcc_tty_driver)) {
		pr_err("VCC: TTY driver alloc failed\n");
		return PTR_ERR(vcc_tty_driver);
	}

	vcc_tty_driver->driver_name = vcc_driver_name;
	vcc_tty_driver->name = vcc_device_node;

	vcc_tty_driver->minor_start = VCC_MINOR_START;
	vcc_tty_driver->type = TTY_DRIVER_TYPE_SYSTEM;
	vcc_tty_driver->init_termios = vcc_tty_termios;
	// 设置操作函数
	tty_set_operations(vcc_tty_driver, &vcc_ops);

	rv = tty_register_driver(vcc_tty_driver);
	if (rv) {
		pr_err("VCC: TTY driver registration failed\n");
		put_tty_driver(vcc_tty_driver);
		vcc_tty_driver = NULL;
		return rv;
	}

	vccdbg("VCC: TTY driver registered\n");

	return 0;
}
```

### 线路(配置)

tty 设备（终端、串口、控制台）的**线路设置**（比如串口的 8N1 模式、是否回显、数据处理方式），用户空间程序**只能通过 `termios.h` 提供的库函数 / 结构体**修改，这是系统规定的标准方法，内核会根据这个配置驱动硬件。

```c
struct termios {
    tcflag_t c_iflag;   // 输入模式标志（控制输入数据怎么处理）
    tcflag_t c_oflag;   // 输出模式标志（控制输出数据怎么处理）
    tcflag_t c_cflag;   // 控制模式标志（硬件核心参数：数据位、停止位、校验位）
    tcflag_t c_lflag;   // 本地模式标志（终端本地行为：回显、行缓冲）
    cc_t     c_cc[NCCS];// 控制字符数组（读取超时、最小读取字节数等）
};
```

+ c_cflag

**直接决定串口通信的底层格式**（嵌入式 / 串口开发最常用）：

| 标志位    | 作用解释                                        |
| --------- | ----------------------------------------------- |
| `CSIZE`   | 数据位掩码（用来设置 5/6/7/8 位数据位）         |
| `CSTOPB`  | 双停止位（不设置 = 1 个停止位，设置 = 2 个）    |
| `PARENB`  | 使能奇偶校验（不设置 = 无校验）                 |
| `PARODD`  | 奇校验（仅`PARENB`开启时有效；不设置 = 偶校验） |
| `CREAD`   | 使能接收字符（关闭则收到数据直接丢弃）          |
| `CRTSCTS` | 硬件流控（CTS/RTS 握手）                        |
| `CLOCAL`  | 忽略调制解调器状态（本地串口直接通信必开）      |

+ c_iflag

| 标志位   | 作用解释                                 |
| -------- | ---------------------------------------- |
| `INPCK`  | 开启奇偶 / 帧错误检查                    |
| `BRKINT` | 收到 Break 信号，清空队列 + 发送中断信号 |
| `PARMRK` | 标记校验 / 帧错误（配合`INPCK`使用）     |
| `IGNPAR` | 忽略所有校验 / 帧错误                    |
| `IGNBRK` | 忽略 Break 信号                          |

+ c_oflag

`OPOST`：**开启输出数据处理**（比如换行转回车、制表符转换），关闭则原生输出

+ c_lflag

| 标志位   | 作用解释                             |
| -------- | ------------------------------------ |
| `ECHO`   | 回显（输入字符显示在屏幕上）         |
| `ICANON` | 正规模式（行缓冲：按回车才读取数据） |
| `ISIG`   | 使能终端信号（Ctrl+C、Ctrl+Z）       |
| `IEXTEN` | 扩展功能                             |

+ c_cc

`VMIN`：最小读取字节数（读串口时，至少收到多少字节才返回）

`VTIME`：读取超时时间（单位：100ms）

#### API函数

```c
// 1. 获取终端当前配置，保存到 termios 结构体
int tcgetattr(int fd, struct termios *termios_p);
// fd：打开的tty/串口文件描述符
// termios_p：传出参数，存储当前配置

// 2. 将结构体中的配置，写入终端硬件
int tcsetattr(int fd, int optional_actions, struct termios *termios_p);
// optional_actions：配置生效时机（最常用 TCSANOW = 立即生效）
```

配置波特率

```c
// 1. 获取【输出波特率】（终端发送数据的速率）
speed_t cfgetospeed(struct termios *termios_p);    

// 2. 获取【输入波特率】（终端接收数据的速率）
speed_t cfgetispeed(struct termios *termios_p);   

// 设置输出波特率
int cfsetospeed(struct termios *termios_p, speed_t speed);
// 设置输入波特率
int cfsetispeed(struct termios *termios_p, speed_t speed);
```

> 大部分最后的处理是在set_termios回调函数里面
