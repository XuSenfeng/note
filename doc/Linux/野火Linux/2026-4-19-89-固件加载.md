# 固件加载

在 Linux 内核中，硬件驱动有时候需要一些由闭源硬件厂商提供的二进制代码（也就是固件 firmware）才能正常工作。内核不应该把这些庞大的二进制文件直接编译到内核源码里，而是使用一套叫做 **Firmware API** 的标准机制。

它基于`firmware_class`子系统实现，负责从各种来源获取硬件设备所需的固件文件，并安全地传递给驱动程序，最终写入设备执行

```c
struct firmware {
	size_t size; // 固件的大小
	const u8 *data; // 指向数据的只读指针

	/* firmware loader private fields */
	void *priv; // 内核使用的私有数据
};
```

Linux 5.10 采用严格的优先级顺序查找固件，一旦找到立即返回：

1. **内置固件**：编译进内核镜像的固件
2. **固件缓存**：内核内存中缓存的已加载固件
3. **直接文件系统查找**：从磁盘指定路径查找
4. **平台固件回退**：从 UEFI 等平台固件中查找嵌入式固件
5. **sysfs 用户空间回退**：通过 sysfs 接口让用户空间工具提供固件Linux [内核文档](https://docs.linuxkernel.org.cn/driver-api/firmware/lookup-order.html?f_link_type=f_linkinlinenote&flow_extra=eyJpbmxpbmVfZGlzcGxheV9wb3NpdGlvbiI6MCwiZG9jX3Bvc2l0aW9uIjowLCJkb2NfaWQiOiI0M2U1Y2UzMjQ4OGFmMjcyLThjZTExZjRhMTY2YzdjMjEifQ%3D%3D)

+ 驱动发起请求

设备驱动检测到硬件需要固件时，调用`request_firmware()`系列函数发起请求。

+ 内置固件检查

内核首先检查是否有同名固件被编译进内核。如果有，直接返回指向该固件的指针，无需访问磁盘。

+ 固件缓存检查

如果固件之前已经加载过且仍在缓存中，直接返回缓存的副本，避免重复加载。

+ 直接文件系统查找

内核按照以下优先级顺序在文件系统中查找固件文件：

> 内核模块参数`firmware_class.path`指定的自定义路径
>
> `/lib/firmware/updates/$(uname -r)/``
>
> ``/lib/firmware/updates/`
>
> `/lib/firmware/$(uname -r)/``
>
> ``/lib/firmware/`

Linux 5.10 原生支持 XZ 压缩的固件文件，会自动尝试查找并解压`.xz`后缀的文件。

+ 平台固件回退

如果驱动使用`firmware_request_platform()`函数，内核会尝试从 UEFI 等平台固件中查找嵌入式固件。这对于某些没有独立固件文件的设备特别有用

+ sysfs 用户空间回退

如果以上方法都失败，内核会在`/sys/class/firmware/`下创建一个以固件名称命名的目录，包含三个关键文件：

- `loading`：控制加载状态 (1 = 开始加载，0 = 加载完成，-1 = 取消加载)
- `data`：用于传递固件数据的二进制接口
- `device`：指向请求固件的设备的符号链接

然后内核发送 uevent 事件通知用户空间。用户空间工具 (如`systemd-udevd`) 接收到事件后，会从其他位置 (如网络、initramfs) 获取固件，并通过 sysfs 接口写入内核

+ 超时处理

如果在指定时间内 (默认 60 秒) 用户空间没有提供固件，内核会取消加载请求并返回错误

+ 驱动使用和释放

驱动获取到固件后，将其写入设备硬件。使用完毕后，必须调用`release_firmware()`函数释放相关内存资源

## API函数

### 同步接口

- `request_firmware()`：最常用的同步加载函数，会阻塞调用者直到加载完成或失败
- `firmware_request_nowarn()`：与`request_firmware()`类似，但加载失败时不输出警告信息
- `request_firmware_direct()`：只进行直接文件系统查找，不触发用户空间回退
- `firmware_request_platform()`：先进行直接文件系统查找，失败后尝试从平台固件中查找
- `request_firmware_into_buf()`：将固件加载到驱动预先分配的缓冲区中Linux Kernel

### 异步接口

- `request_firmware_nowait()`：异步加载固件，不会阻塞调用者。加载完成后通过回调函数通知驱动

### 释放接口

- `release_firmware()`：释放已加载的固件资源

## 示例

 Linux内核里 cx25840 系列（及兼容的编解码芯片如 cx2388x, cx231xx）多媒体驱动的一部分。`cx25840_loadfw` 函数的作用是将固件（Firmware）从主机的宿主文件系统里读取出来，并通过 **I2C 总线**一点点地发送到这颗视频解码芯片内部

```c
int cx25840_loadfw(struct i2c_client *client)
{
	struct cx25840_state *state = to_state(i2c_get_clientdata(client));
	const struct firmware *fw = NULL;
	u8 buffer[FWSEND];
	const u8 *ptr;
	const char *fwname = get_fw_name(client);
	int size, retval;
	int max_buf_size = FWSEND;
	u32 gpio_oe = 0, gpio_da = 0;
	
    // 处理不同芯片的差异点
	if (is_cx2388x(state)) {
		/* Preserve the GPIO OE and output bits */
		gpio_oe = cx25840_read(client, 0x160);
		gpio_da = cx25840_read(client, 0x164);
	}

	/* cx231xx cannot accept more than 16 bytes at a time */
	if (is_cx231xx(state) && max_buf_size > 16)
		max_buf_size = 16;
	
    // 发起固件请求
	if (request_firmware(&fw, fwname, FWDEV(client)) != 0) {
		v4l_err(client, "unable to open firmware %s\n", fwname);
		return -EINVAL;
	}
	// 开始接受固件
	start_fw_load(client);
	
    // 切片发送
	buffer[0] = 0x08;
	buffer[1] = 0x02;

	size = fw->size;
	ptr = fw->data;
	while (size > 0) {
		int len = min(max_buf_size - 2, size);

		memcpy(buffer + 2, ptr, len);

		retval = fw_write(client, buffer, len + 2);

		if (retval < 0) {
			release_firmware(fw);
			return retval;
		}

		size -= len;
		ptr += len;
	}

	end_fw_load(client);

	size = fw->size;
    // 释放固件
	release_firmware(fw);

	if (is_cx2388x(state)) {
		/* Restore GPIO configuration after f/w load */
		cx25840_write(client, 0x160, gpio_oe);
		cx25840_write(client, 0x164, gpio_da);
	}

	return check_fw_load(client, size);
}
```
