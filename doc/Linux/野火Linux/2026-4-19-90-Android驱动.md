# Android驱动

Android的设备驱动与Linux一样，因为Android本身基于Linux内核，但是Android对内核 引入了如下主要补丁

## binder IPC系统

binder机制是Android提供的一种进程间通信方法，使一个进程可以以类似远程过程调用的 形式调用另一个进程所提供的功能, 连接着所有系统服务和应用进程。基于 OpenBinder 项目开发，它解决了传统 Linux IPC 机制在移动设备上性能低、安全性差、开发复杂的问题

它就是一种典型的以miscdevice 形式实现的字符设备，而且提供了一些/proc结点。本质上，binder用户空间的程序绝大多数情况 下在底层是调用了binder驱动的ioctl()函数

Android中的binder通信基于**Service/Client**模型，所有需要IBinder通信的进程都必须创建一 个IBinder接口。Android虚拟机启动之前系统会先启动Service Manager进程，Service Manager 打开binder驱动，并通知binder驱动程序这个进程将作为System Service Manager，然后该进程将 进入一个循环，等待处理来自其他进程的数据

+ **机制层面**：一种高效、安全的跨进程通信机制
+ **驱动层面**：一个 Linux 字符设备驱动 (`/dev/binder`)，作为内核与用户空间的桥梁
+ **框架层面**：连接 Client、Server、ServiceManager 和驱动的完整通信体系
+ **代码层面**：一个实现了`IBinder`接口的类，是所有 Binder 对象的基类

Binder 采用经典的**C/S (客户端 / 服务器) 架构**，由四个核心组件协同工作

| 组件                | 位置                | 核心功能                                                   |
| ------------------- | ------------------- | ---------------------------------------------------------- |
| **Binder 驱动**     | 内核空间            | 进程间数据路由、内存映射管理、线程调度、引用计数、身份验证 |
| **ServiceManager**  | 用户空间 (守护进程) | 全局服务注册与查询中心，维护 "服务名→Binder 引用" 映射表   |
| **Server (服务端)** | 用户空间            | 实现具体业务逻辑，创建 Binder 实体并注册到 ServiceManager  |
| **Client (客户端)** | 用户空间            | 通过 ServiceManager 获取服务代理，发起远程方法调用         |

**Binder 驱动**

- 运行在内核空间，是整个 Binder 机制的核心
- 通过`ioctl()`系统调用与用户空间交互
- 为每个进程维护独立的 Binder 上下文和数据结构
- 自动附加调用者的 UID/PID 信息，用于权限验证

**ServiceManager**

- 系统启动时第一个启动的 Binder 服务
- 拥有固定的 Binder 句柄`0`，是所有服务发现的起点
- 类似 "电话簿"，提供服务的注册和查询功能
- Android 8.0 后引入多个 Binder 域，如`/dev/vndbinder`用于供应商服务

**Binder 实体与引用**

- **Binder 实体 (BBinder)**：服务端进程中实际实现业务逻辑的对象
- **Binder 引用 (BpBinder)**：客户端进程中持有的代理对象，模拟服务端接口
- 驱动负责维护实体与引用之间的映射关系

### 核心机制

+ **一次拷贝机制 (性能核心)**

相比传统 IPC 的两次拷贝，Binder 仅需一次数据拷贝

```bash
Client Process                    Kernel Space                   Server Process
[User]                            [Kernel]                       [User]
┌────────────┐                    ┌────────────┐                 ┌────────────┐
│ 数据       │ ───copy──────────> │ Binder缓冲区│ <───mmap映射─── │ 映射区     │
└────────────┘                    └────────────┘                 └────────────┘
                                      │
                                      ▼
                                同一物理内存
```

1. 服务端进程启动时，通过`mmap()`将内核中的一块 Binder 缓冲区映射到自身用户空间
2. 客户端发起 IPC 调用时，将数据从自己的用户空间**拷贝一次**到内核的 Binder 缓冲区
3. 由于服务端已映射了同一块物理内存，因此可以**直接读取**数据，无需二次拷贝

+ **完整通信流程**

**阶段 1：ServiceManager 初始化**

- 系统启动时，ServiceManager 向 Binder 驱动注册自己
- 驱动为其创建特殊的 Binder 实体，句柄固定为 0
- 所有进程都可以通过句柄 0 访问 ServiceManager

**阶段 2：服务注册**

- 服务端进程创建 Binder 实体 (BBinder)
- 向 ServiceManager 发送注册请求，提供服务名称和 Binder 实体
- ServiceManager 将 "服务名→Binder 引用" 存入注册表

**阶段 3：客户端获取服务代理**

- 客户端通过服务名称向 ServiceManager 发起查询
- ServiceManager 返回对应服务的 Binder 引用 (句柄)
- 客户端将该引用封装为代理对象 (BpBinder)

**阶段 4：远程方法调用**

1. 客户端调用代理对象的方法，参数被打包为`Parcel`对象
2. 代理对象调用`transact()`方法，通过 JNI 进入 Native 层的`IPCThreadState`
3. `IPCThreadState`通过`ioctl()`将请求发送给 Binder 驱动
4. 驱动根据目标句柄找到对应的服务端进程和 Binder 实体
5. 驱动将数据拷贝到内核缓冲区，唤醒服务端的空闲 Binder 线程
6. 服务端的`IPCThreadState`读取数据，交给`onTransact()`方法处理
7. `onTransact()`解析参数，调用实际的业务方法
8. 执行结果被打包为`Parcel`，通过相同路径返回给客户端

## ashemem内存共享机制

**Ashmem (Anonymous Shared Memory，匿名共享内存)** 是 Android 基于 Linux 内核扩展的一套高效共享内存机制，专为移动设备的内存压力与进程间大数据通信设计，核心是 **零拷贝共享 + 内核级内存回收**

> Linux 3.17（2014）引入 **memfd**：
>
> - 匿名、可写、可 seal（锁定不可改）、可通过 fd 共享
> - 标准 POSIX 风格、主线维护、全平台支持
> - Android 12+ 已**用 memfd 替代 Ashmem**

![image](https://picture-01-1316374204.cos.ap-beijing.myqcloud.com/lenovo-picture/202604191533884.png)

### 核心定位与设计目的

- **本质**：内核驱动（`/dev/ashmem`）+ 用户态接口，基于 tmpfs/shmem 封装。
- 解决痛点
    1. **Binder 瓶颈**：Binder 单次事务限 1MB，不适合图像、视频、帧缓冲等大数据。
    2. **传统 IPC 低效**：管道 / Socket 需 2 次拷贝，System V 共享内存无系统级回收。
    3. **移动内存敏感**：支持系统在低内存时**自动回收闲置共享内存**，避免 OOM。
- **核心优势**：**零拷贝**（多进程映射同一段物理内存）、**基于文件描述符（fd）共享**、**可被内核回收**

### 底层原理与关键机制

1. **基本工作流程**

1. **创建**：打开 `/dev/ashmem`，通过 `ioctl` 设置名称、大小 → 得到 **fd**。
2. **映射**：调用 `mmap` 将 fd 映射到进程虚拟地址空间 → 获得可直接读写的指针。
3. **跨进程共享**：通过 **Binder 传递 fd**（内核自动 dup），其他进程同样 mmap 即可访问同一块物理内存。
4. **释放**：`munmap` 解除映射、关闭 fd；所有引用消失后内核自动释放物理页。

2. **核心特性：Pin / Unpin 锁定与回收（最关键设计）**

- **Pin (锁定)**：`ioctl(fd, ASHMEM_PIN, ...)` → 标记内存**正在使用**，内核**不可回收**。
- **Unpin (解锁)**：`ioctl(fd, ASHMEM_UNPIN, ...)` → 标记**暂不使用**，低内存时内核可**换出 / 回收**该页。
- **再访问**：Unpin 后内存若被回收，重新 Pin 会返回 `ASHMEM_WAS_PURGED`，需重新初始化数据。
- **默认**：新创建的 Ashmem 区域默认 **Pin 状态**。

3. **其他重要特性**

- **匿名性**：无 System V 的全局 key，仅靠 fd 传递，更安全、更灵活。
- **延迟分配 (Lazy Allocation)**：`mmap` 时只分配虚拟地址，**首次读写才分配物理页**。
- **权限控制**：`mmap` 时指定 `PROT_READ`/`PROT_WRITE`，可设为只读共享。
- **命名**：`ASHMEM_SET_NAME` 用于调试（`/proc/pid/maps` 可见）。

### 实际使用

```c
// 1. 创建
/*
fd = open("/dev/ashmem", O_RDWR); 
ioctl(fd, ASHMEM_SET_NAME, region_name); 
ioctl(fd, ASHMEM_SET_SIZE, region_size);
*/
int fd = ashmem_create_region("my_shmem", 4*1024*1024);
// 2. 映射
void *ptr = mmap(NULL, 4*1024*1024, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
// 3. 读写
memcpy(ptr, data, size);
// 4. 跨进程传递 fd（Binder）
// 5. 闲置时 Unpin
ashmem_unpin(fd, ptr, 4*1024*1024);
// 6. 再使用前检查
if (ashmem_pin(fd, ptr, len) == ASHMEM_WAS_PURGED) { reinit(); }
// 7. 释放
munmap(ptr, size);
close(fd);
```

## 电源管理

Android电源管理针对标准Linux内核的电源管理进行了一些优化

## OOM内存处理

Linux内核本身提供了OOM（Out Of Memory）机制，它可以在系统内存不够的情况下主动 杀死进程腾出内存。不过Android的Low Memory Killer相对于Linux标准OOM机制更加灵活， 它可以根据需要杀死进程来释放内存

## 日志处理

为了辅助调试，Android增加了用于将内核打印消息保存起来的RAM console和用户应用程 序可以写入、读取log信息的logger设备驱动
