# module

内核里面的模块的加载

**执行一次初始化函数 + 把模块能力挂到内核各框架里，后续靠回调长期工作。**

|          | built-in（`#ifndef MODULE`）          | 可加载模块（`#ifdef MODULE`）                  |
| -------- | ------------------------------------- | ---------------------------------------------- |
| 定义     | `module.h:91` `__initcall(x)`         | `module.h:133` `alias` 到 `init_module`        |
| 调用者   | `do_initcalls()` 遍历 `.initcall6` 段 | `insmod`/`modprobe` 触发内核调用 `init_module` |
| 调用时机 | 内核启动时                            | 模块被加载时                                   |

① 启动时自动加载（built-in）

- 驱动写死进内核，开机就跑
- Kconfig 选 `=y` 时走这条路

② 运行时按需自动加载（built-in 或 .ko 均可）

- 触发条件：打开设备、挂载文件系统、创建 socket 等
- Kconfig 选 `=m` 时，.ko 文件躺在磁盘等待被 request_module 触发

③ 用户手动加载（.ko）

- 完全由用户控制时机

> 所有加载的模块可以在`/proc/modules`文件里面查看
>
> 可以在文件`/proc/kallsyms`查看到所有的导出符号表
>
> 如果一个模块的参数有读写的权限, 出现在对应模块的`/sys/module/{模块名}/parameters`里面看到

```c
	const char *name;
	struct module *mod;
```

### 前置加载

kernel\kmod.c

**内核在运行时发现需要某个模块，通过用户态的 `modprobe` 工具把它动态加载进来**，实现"按需加载"而不是启动时全部加载。内核里面会调用这个请求某一个模块


## 初始化

初始化的时候有两种, 一个是编译到内核里面, 一种是编译为.ko模块

### 编译到内核

```c
/**
 * module_init() - driver initialization entry point
 * @x: function to be run at kernel boot time or module insertion
 *
 * module_init() will either be called during do_initcalls() (if
 * builtin) or at module insertion time (if a module).  There can only
 * be one per module.
 */
#define module_init(x)	__initcall(x);
#define __initcall(fn) device_initcall(fn)
#define __define_initcall(fn, id) ___define_initcall(fn, id, .initcall##id)
#define ___define_initcall(fn, id, __sec)			\
	__unique_initcall(fn, id, __sec, __initcall_id(fn))
#define __unique_initcall(fn, id, __sec, __iid)			\
	____define_initcall(fn,					\
		__initcall_stub(fn, __iid, id),			\
		__initcall_name(initcall, __iid, id),		\
		__initcall_section(__sec, __iid))
```

把函数指针（或相对重定位项）放进特定的 `.initcall*` 链接段, 启动时由内核统一遍历这些段并调用, 不同的类型的`initcall`的等级不一样

所有默认加载的模块统一在函数`do_initcalls`函数里面执行

在链接文件里面

```lds
#define INIT_CALLS							\
		__initcall_start = .;					\
		KEEP(*(.initcallearly.init))				\
		INIT_CALLS_LEVEL(0)					\
		INIT_CALLS_LEVEL(1)					\
		INIT_CALLS_LEVEL(2)					\
		INIT_CALLS_LEVEL(3)					\
		INIT_CALLS_LEVEL(4)					\
		INIT_CALLS_LEVEL(5)					\
		INIT_CALLS_LEVEL(rootfs)				\
		INIT_CALLS_LEVEL(6)					\
		INIT_CALLS_LEVEL(7)					\
		__initcall_end = .;
```

执行的时候按照待机依次执行这些函数

1. 你写模块源码（比如 `foo.c`），只写 `module_init` / `module_exit` / `MODULE_LICENSE` 这类宏。
2. 编译时 `modpost` 会生成 `foo.mod.c`。
3. 在这个自动生成文件里，add_header()会写出：
    - `__visible struct module __this_module`
    - 并放进 `.gnu.linkonce.this_module` 节。代码就在`scripts\mod\modpost.c`的add_header函数里面。
4. 最终链接成 `foo.ko` 后，内核加载器在加载时用
    `info->index.mod = find_sec(info, ".gnu.linkonce.this_module");`
    去找到这节，见 module.c:3278。

### 编译ko文件

```c
#define module_init(initfn)					\
	static inline initcall_t __maybe_unused __inittest(void)		\
	{ return initfn; }					\
	int init_module(void) __copy(initfn) 			\
		__attribute__((alias(#initfn)));		\
	__CFI_ADDRESSABLE(init_module)
```

这是一个类型检查, 不会生成实际的代码,函数签名不对，编译器会直接报错

```bash
static inline initcall_t __maybe_unused __inittest(void)
{ return initfn; }
```

给这个函数创建一个别名,`alias(#initfn)` 把 `init_module` 定义为 `initfn`的**符号别名**（不是函数调用，是链接器层面的同名）, `__copy(initfn)` 把 `initfn` 的函数属性（如 `__init`）复制给 `init_module`

```c
int init_module(void) __copy(initfn)
    __attribute__((alias(#initfn)));
```

### 信息记录

#### 信息

记录在elf文件`.modinfo`里面的用户信息, 在加载的时候作为配置参数

```c
/* Generic info of form tag = "info" */
// 创建一个唯一名字的字符串, __used表示不要优化, 记录在.modinfo字段
// 最后是实际存储的字符的拼接
#define MODULE_INFO(tag, info) __MODULE_INFO(tag, tag, info)
#define __MODULE_INFO(tag, name, info)					  \
static const char __UNIQUE_ID(name)[]					  \
  __used __section(".modinfo") __attribute__((unused, aligned(1)))	  \
  = __MODULE_INFO_PREFIX __stringify(tag) "=" info

// __COUNTER__是编译器预处理器提供的扩展宏。
// 每展开一次，返回一个递增整数（如 0,1,2...），同一翻译单元内单调增加
// 先拼前缀，再拼当前计数值，得到唯一变量名
// __UNIQUE_ID_##tag##__COUNTER__
#define ___PASTE(a,b) a##b
#define __PASTE(a,b) ___PASTE(a,b)
#define __UNIQUE_ID(prefix) __PASTE(__PASTE(__UNIQUE_ID_, prefix), __COUNTER__)

// 是把宏参数转成字符串字面量的宏
// 这里的#是把一个序列转换为字符串, 多了一层宏定义, 在处理的时候, 如果参数是
// 一个宏定义, 回合__stringify一起展开一次, 再变为字符串
#define __stringify_1(x...)	#x
#define __stringify(x...)	__stringify_1(x)
```

#### 参数

实际是把这个参数的地址, 操作函数等记录在`__param`字段里面, 会记录在sysfs里面, 可以被其他的module调用

```c
static char ctype=0;
module_param(ctype,byte,0);

#define module_param(name, type, perm)				\
	module_param_named(name, name, type, perm)

#define module_param_named(name, value, type, perm)			   \
	param_check_##type(name, &(value));				   \
	module_param_cb(name, &param_ops_##type, &value, perm);		   \
	__MODULE_PARM_TYPE(name, #type)

#define module_param_cb(name, ops, arg, perm)				      \
	__module_param_call(MODULE_PARAM_PREFIX, name, ops, arg, perm, -1, 0)

#define __module_param_call(prefix, name, ops, arg, perm, level, flags)	\
	/* Default value instead of permissions? */			\
	static const char __param_str_##name[] = prefix #name;		\
	static struct kernel_param __moduleparam_const __param_##name	\
	__used								\
    __section("__param") __attribute__ ((unused, aligned(sizeof(void *)))) \
	= { __param_str_##name, THIS_MODULE, ops,			\
	    VERIFY_OCTAL_PERMISSIONS(perm), level, flags, { arg } }
```

+ 操作

```c
STANDARD_PARAM_DEF(byte,	unsigned char,		"%hhu",		kstrtou8);
```

## 编译过程

不同的module实际是走的`module.h`文件里面的`#define MODULE`分支

1. **编译源文件为 `.o`**Kbuild 按普通内核规则编译各个 `*.c`，但会带上 `-DMODULE`。添加多一个宏定义
2. **把模块内多个对象先做一次“合并链接”**例如把 `a.o b.o` 链成 `xxx.o`
3. `modpost` 扫描模块并生成 `xxx.mod.c`modpost 会做：
    - 未导出符号检查、GPL 符号检查
    - 生成符号版本信息（`CONFIG_MODVERSIONS` 时）
    - 生成模块元信息代码（`struct module`、`MODULE_INFO`、`vermagic` 等）
        相关代码在 modpost.c，规则在 Makefile.modpost。
4. **编译 `xxx.mod.c` -> `xxx.mod.o`**
5. 最终链接生成 `xxx.ko`把 `xxx.o + xxx.mod.o` 再链接成最终可加载模块 `xxx.ko`。规则在 Makefile.modfinal。
6. （可选）**BTF / 模块签名**
    - `CONFIG_DEBUG_INFO_BTF_MODULES`：生成模块 BTF
    - CONFIG_MODULE_SIG：对 `.ko` 签名

### modpost文件

这个文件是在module编译为.ko的时候使用的, 会被编译为modpost工具, 这个时候内核文件已经被编译为.o文件了, 使用modpost处理.o文件, .o文件也是一种elf文件

1. **检查**

- 扫 ELF 符号与重定位
- 检查未解析符号、GPL-only 符号使用、命名空间导入
- 检查 section mismatch（[__init/__exit](vscode-file://vscode-app/d:/vscode/Microsoft VS Code/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 引用错误）

2. **生成**

- 为每个模块生成 `xxx.mod.c`
- 在里面生成 __this_module（`struct module`），放进 `.gnu.linkonce.this_module`
- 生成 MODULE_INFO(...)、depends、srcversion、版本相关信息

3. **维护符号版本**

- 读取/写出 `Module.symvers`
- 处理 `__versions` CRC（`CONFIG_MODVERSIONS`）

## 地址同步

1. 先把 section 拷到最终地址（`core_layout.base` / `init_layout.base`），并更新 `sh_addr`
2. `simplify_symbols()` 把符号值转成直接可用地址（内部符号用 section 的 `sh_addr` + offset，外部符号通过 `resolve_symbol_wait()` 找到）
3. `apply_relocations()` 把这些地址打补丁到代码/数据里
4. 后续内核触发回调时，直接用已填好的函数指针
