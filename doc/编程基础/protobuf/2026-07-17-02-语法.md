# 语法

https://protobuf.com.cn/programming-guides/proto3/#scalar

定义一个搜索请求消息格式，其中每个搜索请求都有一个查询字符串、您感兴趣的特定结果页面以及每页的结果数量。

```protobuf
syntax = "proto3";

message SearchRequest {
  string query = 1;
  int32 page_number = 2;
  int32 results_per_page = 3;
}
```
+ 文件的第一行指定您使用的是 protobuf 语言规范的 proto3 版本， 必须是文件的第一个非空、非注释行。

>其他可以使用的还有`edition = "2023";`或者`syntax = "proto2";`
>如果没有这部分, 默认使用的是proto2

- `SearchRequest` 消息定义指定了三个字段（名称/值对），每种你想包含在此类消息中的数据都对应一个字段。每个字段都有一个名称和一个类型。


一旦消息类型投入使用，这个编号就**不能更改**, 字段编号**永远不应被重用**。
##  消息字段

你必须为消息定义中的每个字段赋予一个介于 `1` 和 `536,870,911` 之间的编号

+ 给定的编号**必须**在该消息的所有字段中是**唯一的**。
+ 字段编号 `19,000` 到 `19,999` 为 Protocol Buffers **实现保留**。如果你在消息中使用这些保留的字段编号，protocol buffer 编译器会报错。
+ 您不能使用任何先前保留的字段编号或已分配给扩展的任何字段编号。

### 保留

当您不再需要某个字段且所有引用都已从客户端代码中删除时，您可以从消息中删除该字段定义。但是，您**必须保留**已删除的字段编号。

如果您通过完全删除字段或将其注释掉来更新消息类型，未来的开发人员在对该类型进行自己的更新时可以重用该字段编号。这可能会导致严重问题，如重用字段编号的后果中所述。为确保这种情况不会发生，请将您删除的字段编号添加到 `reserved` 列表中

```protobuf
message Foo {
  reserved 2, 15, 9 to 11;
}
```

### 基数

基数决定一个字段能出现多少次、序列化规则、能不能区分「没赋值」和「赋值成默认值」

proto3 一共有四类基数：**`optional`、隐式默认字段、`repeated`、`map`**。


1. `optional`

```protobuf
optional int32 age = 1; 
optional User info = 2;
```

设置的时候, 不管值是不是默认值（比如 `age=0`），都会序列化到二进制；代码里可以判断「此字段被显式赋值」
没有设置的时候, 不会序列化；读取时返回类型默认值；代码可以检测「有没有设置过」。
> 可以区分：**没设置 vs 设置了默认值**
2. 隐式字段, 不设置
```protobuf
int32 age = 1; // 基础类型，隐式基数 
User info = 2; // 消息类型，隐式基数
```

实际的处理分为两种

+ 设置的是一个**消息类型**, 行为**完全等价于 optional**
+  设置的是一个**基础类型**, 值不等于默认值的时候才进行序列化, 接收方无法分辨你到底是主动赋值默认值，还是压根没传这个字段
3. `repeated`可以重复的
```protobuf
repeated string tags = 3; 
repeated Order orders = 4;
```
实际就是一个数组
- 字段可以出现 **0 次或多次**（数组 / 列表）
- 顺序保留，序列化后解析出来顺序不变
- 哪怕是空列表：    
    - protobuf 二进制里**不会编码空数组**
    - 解析时得到空列表
- 列表内允许重复元素

4. `map`映射字段

```protobuf
map<string, int32> score_map = 5;
```

本质是语法糖，等价于：

```protobuf
repeated MapEntry entries;
message MapEntry {
  optional string key = 1;
  optional int32 value = 2;
}
```

- Key 只能是基础标量类型（不能是 message）
- key 唯一，不存在重复 key
- **不保证顺序**（不要依赖遍历顺序）
- 空 map 不会序列化到二进制
- 等价于编程语言里的字典、HashMap

## 注释

- 优先使用 C/C++/ [Java](https://protobuf.com.cn/programming-guides/proto3/#) 风格的行尾注释 '//'，放在 .proto 代码元素的前一行。
- 也接受 C 风格的内联/多行注释 `/* ... */`。
    - 使用多行注释时，推荐使用 '\*' 作为边距行。

### 默认值

- 对于字符串，默认值是空字符串。
- 对于字节，默认值是空字节。
- 对于布尔值，默认值是 false。
- 对于数值类型，默认值是零。
- 对于消息字段，该字段未设置。其确切值取决于语言。详情请参阅[生成的代码指南](https://protobuf.com.cn/reference/)。
- 对于枚举，默认值是**第一个定义的枚举值**，它必须为 0。
### 枚举

当您定义消息类型时，您可能希望它的某个字段只能是预定义列表中的值之一。例如，假设您想为每个 `SearchRequest` 添加一个 `corpus` 字段，其中 corpus 可以是 `UNIVERSAL`、`WEB`、`IMAGES`、`LOCAL`、`NEWS`、`PRODUCTS` 或 `VIDEO`。您可以通过在您的消息定义中添加一个 `enum` 并为每个可能的值定义一个常量来非常简单地实现这一点。

```protobuf
enum Corpus {
  CORPUS_UNSPECIFIED = 0;
  CORPUS_UNIVERSAL = 1;
  CORPUS_WEB = 2;
  CORPUS_IMAGES = 3;
  CORPUS_LOCAL = 4;
  CORPUS_NEWS = 5;
  CORPUS_PRODUCTS = 6;
  CORPUS_VIDEO = 7;
}

message SearchRequest {
  string query = 1;
  int32 page_number = 2;
  int32 results_per_page = 3;
  Corpus corpus = 4;
}
```
在枚举值加前缀时，剥离前缀后的其余名称仍应是合法且符合样式的枚举名称
```protobuf
enum DeviceTier {
  DEVICE_TIER_UNKNOWN = 0;
  DEVICE_TIER_1 = 1;
  DEVICE_TIER_2 = 2;
}
```
其中 `DEVICE_TIER_` 部分被视为对枚举值进行作用域限定，而不是作为单个枚举值名称的一部分。一些 Protobuf 实现会自动剥离与包含枚举名称匹配的前缀（在安全的情况下），但在本例中不能，因为裸露的 `1` 不是合法的枚举值名称
在 proto3 中，枚举定义中定义的第一个值**必须**为零，并且应该命名为 `ENUM_TYPE_NAME_UNSPECIFIED` 或 `ENUM_TYPE_NAME_UNKNOWN`。这是因为

- 必须有一个零值，以便我们可以将 0 用作数字默认值。
- 零值需要是第一个元素，以便与 proto2 语义兼容，其中除非明确指定了不同的值，否则第一个枚举值是默认值。

## 使用其他消息类型

您可以使用其他消息类型作为字段类型。例如，假设您希望在每个 `SearchResponse` 消息中包含 `Result` 消息——为此，您可以在同一个 `.proto` 文件中定义一个 `Result` 消息类型，然后在 `SearchResponse` 中指定一个类型为 `Result` 的字段

```protobuf
```proto
message SearchResponse {
  repeated Result results = 1;
}

message Result {
  string url = 1;
  string title = 2;
  repeated string snippets = 3;
}
```
