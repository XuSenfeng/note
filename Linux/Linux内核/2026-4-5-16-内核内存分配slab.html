内核内存

2026-04-18
编辑本页

内存分配

  • kmalloc: 分配的是连续的, 最小的是4k, 申请的内存位于物理内存映射区域，而且在物理上也是连续的，它们与真实的物理地址只有一个固定的偏移
  • vmalloc: 分配的是不连续的, 在虚拟内存空间给出一块连续的内存区，实质上，这片连续的虚拟内存在物理内存中并不一定连续

伙伴系统

分配器 设计风格 / 目标 典型适用场景
SLAB 早期 Solaris 风格，每个 CPU 有对象队列，强调“对象缓存 + 高性能”，元数据多、实现复杂 传统服务器/桌面内核；现在基本只是保留兼容，新系统不建议再用
SLOB 基于 K&R 简单分配器思路，尽量少用额外元数据，极致省内存 极小内存（几 MB）的嵌入式设备；从 6.2 起已标记为 deprecated，计划移除
SLUB “无队列（unqueued）”设计，尽量少队列、少锁，强调可扩展性、NUMA 感知、调试便利 自 2.6.23 起成为默认分配器，适合服务器/多核/大内存；是官方推荐的后端

如果内核每次只要 1 字节内存，就去物理内存里找，那物理内存很快就会变成马蜂窝（严重碎片化）, 使用buddy算法进行管理

Linux 管理物理内存时，使用的是“伙伴系统”。它把连续的物理内存页（每页默认 4KB）按 2的n次方 打包成块。 那一串数字，从左到右分别对应 order 0 到 order 10（也就是20到 210个连续页）。

(base) jiao@jiao-virtual-machine:~$ cat /proc/buddyinfo 
Node 0, zone      DMA      0      0      0      1      2      1      1      0      1      1      3 
Node 0, zone    DMA32      2      1      2      1      3      1      3      4      2      4    744 
Node 0, zone   Normal      2      2    166    126     74     38      7      3      3      2   2761
  • Node 0：代表 NUMA 节点。普通的个人电脑或单路虚拟机通常只有一个节点，就是 Node 0。
  • zone DMA / DMA32 / Normal：代表内存区域。
    • DMA：非常古老的外设用的低端内存（通常前16MB）。
    • DMA32：支持32位DMA的设备使用的内存（通常前4GB）。
    • Normal：内核和常规用户程序使用的常规内存（你平时跑程序主要用的就是这里）。

系统刚刚启动不久，或者内存经过回收，大量连续的物理内存被合并成了最大的 4MB 块

如果一台机器跑了很久，经常分配释放内存，数据可能会变成：1500 800 20 2 0 0 0 0 0 0 0。 这就意味着：虽然系统总空闲内存还有很多（第一列加起来不少），但是它们在物理地址上是不连续的！如果这时候有个程序非要申请 2MB 的连续物理内存（比如网卡驱动或者数据库），就会直接报错（OOM），这就是典型的外部碎片化

  • 规则：伙伴系统把连续的页框按 2n（1, 2, 4, 8, 16...页）打包成“块”。
  • 分配：当内核需要 8 个页（32KB）时，它直接去 Order 3 的链表里拿一个现成的块。
  • 分裂：如果 Order 3 空了，它就会去 Order 4 拿一个 16页的块，劈成两半（两个8页的块），一半给出去，一半放到 Order 3 备用。

slab缓存

(87 封私信 / 80 条消息) Linux 内核 | 内存管理——slab 分配器 - 知乎

用于管理比较小块的内存, 内存池, 分配大量小对象的时候, 建立一个二级缓存机制, 首先分配一个内存池, 之后在内存池里面分配小的数据, 在伙伴系统之上架设了 Slab 分配器。

  • 机制：Slab 先向“批发商”（伙伴系统）申请一整页（4KB），然后把这一页像切蛋糕一样切成很多固定大小的小块。比如专门切出 32字节 一块的区域，专门切出 64字节 一块的区域。
  • 用途：内核中频繁创建销毁的对象（如进程描述符 task_struct、文件描述符 inode）都从这里分配。它就像个零件盒子，极大地节省了内核内存，并提高了分配速度。

可以使用命令sudo cat /proc/slabinfo查看, 这里面的数据有专门适配某个结构的, 也有通用的

API

分配使用的是kmlloc

{
    if (__builtin_constant_p(size)) {
#ifndef CONFIG_SLOB
    	unsigned int index;
#endif
        // 分配大内存
    	if (size > KMALLOC_MAX_CACHE_SIZE)
    		return kmalloc_large(size, flags);
#ifndef CONFIG_SLOB
    	index = kmalloc_index(size);

    	if (!index)
    		return ZERO_SIZE_PTR;

    	return kmem_cache_alloc_trace(
    			kmalloc_caches[kmalloc_type(flags)][index],
    			flags, size);
#endif
    }
    // slab处理小内存
    return __kmalloc(size, flags);
}
// in mm/slab.c
void *__kmalloc(size_t size, gfp_t flags)
{
    return __do_kmalloc(size, flags, _RET_IP_);
}
static __always_inline void *__do_kmalloc(size_t size, gfp_t flags,
                      unsigned long caller)
{
    struct kmem_cache *cachep;
    void *ret;

    if (unlikely(size > KMALLOC_MAX_CACHE_SIZE))
        return NULL;
    // 实际分配slab的位置
    cachep = kmalloc_slab(size, flags);
    if (unlikely(ZERO_OR_NULL_PTR(cachep)))
        return cachep;
    // 分配一个slab数据
    ret = slab_alloc(cachep, flags, caller);

    ret = kasan_kmalloc(cachep, ret, size, flags);
    trace_kmalloc(caller, ret,
              size, cachep->size, flags);

    return ret;
}

struct kmem_cache *kmalloc_slab(size_t size, gfp_t flags)
{
    unsigned int index;

    if (size <= 192) {
        // 在分配小于 192 的 obj 时，会向上对齐，例如，9~16 字节的 obj
        // 保存到 16 字节的kmem_cache中，65-96字节的 obj 保存到 
        // 96字节的 mem_cache中。
        if (!size)
            return ZERO_SIZE_PTR;
    	// 使用size获取一个index
        // 映射到 kmalloc_caches 中对应的缓存索引, 
        // 因为有一部分非2的整数次幂大小
        // 这些大小不能用 fls(size-1) 直接得到，
        // 所以需要 size_index[] 这种表驱动映射
        index = size_index[size_index_elem(size)];
    } else {
        if (WARN_ON_ONCE(size > KMALLOC_MAX_CACHE_SIZE))
            return NULL;
        // fls()函数，其作用是得出 最高有效位（二进制下，
        // 最低向高位，最后一个 1的位置）
        index = fls(size - 1);
    }

    return kmalloc_caches[kmalloc_type(flags)][index];
}

static inline unsigned int size_index_elem(unsigned int bytes)
{
    return (bytes - 1) / 8;
}

数据类型

由于slab的缓存特性，slab 分配器从 buddy 分配器中获取的物理内存称为 内存缓存（与 CPU 的硬件缓存进行区别，下文都称为缓存），使用结构体struct kmem_cache（定义在include/linux/slab_def.h文件中）描述。

struct kmem_cache {
/* 2) touched by every alloc & free from the backend */
    slab_flags_t flags;     /* constant flags */
    unsigned int num;       /* 每个 slab 中的 objs 数量 */
/* 3) cache_grow/shrink */
    /* 每个 slab 所使用的页框数量 order */
    unsigned int gfporder;
    /* force GFP flags, e.g. GFP_DMA */
    gfp_t allocflags;
    size_t colour;          /* cache colouring range */
    unsigned int colour_off;    /* colour offset */
    struct kmem_cache *freelist_cache;
    unsigned int freelist_size;
/* 4) cache creation/removal */
    const char *name;
    struct list_head list;
    int refcount;
    int object_size;
    int align;
    struct kmem_cache_node *node[MAX_NUMNODES];
};

SLAB分配器由可变数量的缓存组成，这些缓存由称为“缓存链”的双向循环链表链接在一起（如下图中的 kmem_cache 链表）

struct kmem_cache 结构描述的一段内存就称作一个 slab 缓存池

在slab分配器的上下文中，缓存是特定类型的多个对象的管理器，例如使用cat /proc/slabinfo命令输出的mm_struct 或fs_cache缓存，其名字保存在kmem_cache->name中（Linux 支持单个最大的 slab 缓存大小为32MB ）。kmem_cache 中所有对象的大小是相同的(object_size)，并且此 kmem_cache 中所有SLAB的大小也是相同的(gfporder、num)

每个缓存节点在内存中维护称为slab的连续页块，这些页面被切成小块，用于缓存数据结构和对象。 kmem_cache的 kmem_cache_node 成员记录了该kmem_cache 下的所有 slabs 列表

kmem_cache_node 定义在mm/slab.h文件中

kmem_cache_noed 记录了3种slab：

  • slabs_full ：已经完全分配的 slab
  • slabs_partial： 部分分配的slab
  • slabs_free：空slab，或者没有对象被分配

以上3个链表保存的是slab 描述符，Linux kernel 使用 struct page 来描述一个slab。单个slab可以在slab链表之间移动，例如如果一个半满slab被分配了对象后变满了，就要从 slabs_partial 中被删除，同时插入到 slabs_full 中去

只有在申请对象时发现没有可用的 slab 时才会创建一个新的SLAB，并加入到这三个链表中的一个中。

实际的分配

SLAB

SLUB

  1. 查找空闲页：SLUB 先去当前 CPU 的局部缓存找，如果没有，就去全局的 partial 链表找。
  2. 解引用：拿到 partial 链表中的一个 struct page。
  3. 身份确认：通过 page->slab_cache 确认，没错，这是装 task_struct 的页。
  4. 取出对象：查看 page->freelist，发现它指向地址 0xFFFF_AAAA_0000（假设是页内的第一个对象）。
  5. 更新指针：把 0xFFFF_AAAA_0000 这个地址里存的内容读出来（里面存着下一个空闲对象 0xFFFF_AAAA_0190），把它赋值给 page->freelist。
  6. 更新状态：通过位操作，把 page->counters 里的 inuse 计数器 +1。
  7. 判断状态：如果 inuse 等于总对象数，说明这个页满了，通过 page->next 把它从 partial 链表摘除，挂到 full 链表。
  8. 返回：把 0xFFFF_AAAA_0000 返回给内核，内核把它当做 task_struct 开始初始化。

kmem_cache_init()函数在初始化过程中调用create_boot_cache()对 kmem_cache_boot进行了进一步初始化。第一个kmem cache实例用于为创建其他kmem cache实例分配空间，其大小与系统内的 node 有关。offsetof求出 kmem_cache 结构体中 node的偏移地址，再加上系统内 node 的个数乘以sizeof(struct kmem_cache_node *)

// slab.c
void __init kmem_cache_init(void)
{
    int i;
    // 设置kmem_cache指向静态分配的启动缓存
    kmem_cache = &kmem_cache_boot;

    // NUM_INIT_LISTS 为 2 * 系统内的节点数量
    // 初始化 kmem_cache_node 结构体
    for (i = 0; i < NUM_INIT_LISTS; i++)
        kmem_cache_node_init(&init_kmem_cache_node[i]);
    // 完成第一个kmem cache实例kmem_cache的初始化
    // 第一个kmem cache实例用于为创建其他kmem cache实例分配空间
    // 这里使用offsetof计算实际的大小
    create_boot_cache(kmem_cache, "kmem_cache",
        offsetof(struct kmem_cache, node) +
                  nr_node_ids * sizeof(struct kmem_cache_node *),
                  SLAB_HWCACHE_ALIGN, 0, 0);
    // kmem cache实例加入slab_caches链表
    // slab_caches 是一个全局链表头，用来管理系统中所有的SLAB缓存
    list_add(&kmem_cache->list, &slab_caches);
    slab_state = PARTIAL;

    // 用于分配 kmem_cache_node 结构体
    // 存储到 kmalloc_caches 缓存数组中, KMALLOC_NORMAL - 普通内存分配类型
    // INDEX_NODE - 计算出来的这个大小的数据的位置
    kmalloc_caches[KMALLOC_NORMAL][INDEX_NODE] = create_kmalloc_cache(
                kmalloc_info[INDEX_NODE].name[KMALLOC_NORMAL],
                kmalloc_info[INDEX_NODE].size,
                ARCH_KMALLOC_FLAGS, 0,
                kmalloc_info[INDEX_NODE].size);
    slab_state = PARTIAL_NODE;
    // 建立 kmalloc 缓存大小与索引的映射表, 
    // 让后续 kmalloc 调用能快速找到合适的缓存
    setup_kmalloc_cache_index_table();

    slab_early_init = 0;

    /* 5) Replace the bootstrap kmem_cache_node */
    {
        int nid;
    	//遍历系统中每一个在线的NUMA节点, 
        // NUMA = Non-Uniform Memory Access（非一致性内存访问）
        // 在单核系统只有1个节点，多核服务器可能有多个节点
        for_each_online_node(nid) {
            // 为 kmem_cache 缓存初始化该节点的缓存链表
            init_list(kmem_cache, &init_kmem_cache_node[CACHE_CACHE + nid], nid);

            init_list(kmalloc_caches[KMALLOC_NORMAL][INDEX_NODE],
                      &init_kmem_cache_node[SIZE_NODE + nid], nid);
        }
    }
    // 最后创建所有其他大小的 kmalloc 缓存（8、16、32...字节）
    create_kmalloc_caches(ARCH_KMALLOC_FLAGS);
}

kmalloc_caches: kmem_cache的二维数组，存放所有 kmalloc 缓存, 使用 kmalloc_info 中保存的信息生成不同 size 的 kmem_cache

  • 第一维 [NR_KMALLOC_TYPES]- 缓存类型（普通、回收、DMA等）
  • 第二维 [KMALLOC_SHIFT_HIGH + 1] - 不同大小的缓存（8字节、16字节、32字节... 直到最大值）

传统的单核/小规模多核系统，所有CPU访问内存的速度都一样（UMA）。但在大规模多核服务器中, 为了性能，需要每个缓存在每个节点上

array_cache是kmem_cache 里面的一个成员, 用来为每个CPU维护一个空间对象链表

struct array_cache {
    // 可以使用的对象的数量
    unsigned int avail;
    // 可以拥有的最大的对象的数量
    unsigned int limit;
    // 要从 slab_list 转移进本地高速缓存对象的数量，
    // 或从本地高速缓存中转移出去的 obj 数量
    unsigned int batchcount;
    // 是否在收缩后访问过
    unsigned int touched;
    // 伪数组，保存释放的对象指针
    void *entry[];
};

内核为每个CPU维护一个这样的链表，当需要新的对象时，会优先尝试从当前CPU的本地CPU空闲对象链表获取相应大小的对象，这样就能更快地分配对象

本地CPU空闲对象链表在系统初始化完成后是一个空的链表，只有释放对象时才会将对象加入这个链表。链表对象个数是有限制的

对象释放流程如下：

首先会先将对象释放到本地CPU空闲对象链表中，如果本地CPU空闲对象链表中对象过多，kmem_cache 会将本地CPU空闲对象链表中的batchcount个对象移动到所有CPU共享的空闲对象链表链表中， - 如果所有CPU共享的空闲对象链表链表的对象也太多了，kmem_cache也会把所有CPU共享的空闲对象链表链表中batchcount个数的对象移回它们自己所属的SLAB中， - 这时如果SLAB中空闲对象太多，kmem_cache会整理出一些空闲的SLAB，将这些SLAB所占用的页框释放回页框分配器中

虚拟内存malloc

C 语言调用 malloc() 或者 C++ 调用 new 时发生

用户程序是绝对不能直接接触物理内存的（否则一个烂程序就能把系统搞崩）。Linux 给每个程序都画了一张“大饼”——虚拟内存空间（比如 64位系统下，每个程序以为自己有 128TB 内存）

  1. glibc 接手：标准库的 malloc 并不会马上找内核要内存。
  2. 小内存分配：如果申请的内存较小（通常 < 128KB），glibc 会使用 brk() 系统调用，把程序内存里的“堆”顶指针往上挪一挪。这块虚拟内存依然没有对应真实的物理内存。
  3. 大内存分配：如果申请的内存很大（>= 128KB），glibc 会使用 mmap() 系统调用，在虚拟内存的空白处划出一块新区域。
