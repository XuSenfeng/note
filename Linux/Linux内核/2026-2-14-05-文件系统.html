文件系统

2026-02-26
编辑本页

是磁盘管理的目录, 也可以操作所有硬件和系统的功能机制

initramfs 场景（现代主流）：临时根文件系统上执行的第一个程序为 /init，负责加载驱动、准备并挂载真实根文件系统

传统 initrd 场景：临时根文件系统上执行的第一个程序为 /linuxrc，完成准备工作后退出，内核再执行真实根文件系统的 /sbin/init

无临时根文件系统场景：内核直接挂载真实根文件系统，依次查找执行 /sbin/init → /etc/init → /bin/init → /bin/sh

在根文件系统里面有各种的库, 程序, 以及配置文件, 各种设备节点

linuxrc文件

可以在busybox的代码里面找到这个程序, init\init.c

  1. 内核启动后执行 /sbin/init，而该路径是 BusyBox（或指向 BusyBox 的链接）；
  2. initrd 环境下内核执行 /linuxrc（同样指向 BusyBox）；
  3. 运行 busybox init … 或通过符号链接/硬链接以 init名称调用 BusyBox。
  • 它负责设置终端、环境变量、信号处理，并根据 inittab 或默认规则启动和监管其他进程
  • 定义了可用的 init 动作类型（sysinit/respawn/askfirst/wait/once 等）及其名称映射，后续解析 inittab 时使用
  • 维护一个链表 init_action_list，保存要执行的动作（命令、TTY、类型、PID），并提供添加/删除函数
  • 读取 /etc/inittab文件（或使用内置默认动作）并填充动作列表，默认会添加 sysinit/askfirst shell、关机/重启处理等, 这个文件是一个配置文件, 默认会设置一些状态下面的运行命令
  • 安装信号处理（关机、重启、CAD、SIGHUP 重新加载 inittab），设置环境/控制台，解析 inittab，依次运行 SYSINIT→WAIT→ONCE，然后在主循环中持续处理 RESPAWN/ASKFIRST 并回收子进程
  • 运行/etc/init.d/rcS文件里面的程序(初始化的最初), 之后是WAIT和ONCE的, 然后循环处理RESPAWN和ASKFIRST的程序

设置了一系列的初始化命令, 之后依次执行命令, 最后等待循环命令栏退出

  • 解析 inittab（或默认规则）构建一组 init 动作（命令+类型+TTY）。
  • 启动阶段依次执行 SYSINIT→WAIT→ONCE。
  • 进入循环后反复处理 RESPAWN/ASKFIRST，并 wait 子进程退出，退出的会被标记并重启。

所以它既“按顺序执行初始化命令”，也“常驻等待子进程退出并重启”。如果你说的“等待某两款退出”是指等待子进程退出，这里是等待所有子进程的退出事件。

内核的使用

在linux3.1.2里面init\main.c, 查找文件系统里面的init程序, start_kernel==>rest_init==>kernel_init==>init_post

/*
 * We try each of these until one succeeds.
 *
 * The Bourne shell can be used instead of init if we are
 * trying to recover a really broken machine.
 */
if (execute_command) { // 执行自定义的命令
    run_init_process(execute_command);
    printk(KERN_WARNING "Failed to execute %s.  Attempting "
                "defaults...\n", execute_command);
}
// 执行默认的程序
run_init_process("/sbin/init");
run_init_process("/etc/init");
run_init_process("/bin/init");
run_init_process("/bin/sh");
// 没有可以执行的, 内核启动失败
panic("No init found.  Try passing init= option to kernel. "
      "See Linux Documentation/init.txt for guidance.");

这里使用的参数execute_command是在init_setup函数里面配置的, 是uboot传进来的, init=xxxx的一个参数

static int __init init_setup(char *str)
{
    unsigned int i;

    execute_command = str;
    /*
     * In case LILO is going to boot us with default command line,
     * it prepends "auto" before the whole cmdline which makes
     * the shell think it should execute a script with such name.
     * So we ignore all arguments entered _before_ init=... [MJ]
     */
    for (i = 1; i < MAX_INIT_ARGS; i++)
    	argv_init[i] = NULL;
    return 1;
}
__setup("init=", init_setup);

使用的宏定义__setup会把这个函数记录在.init.setup段里面, 记录为obs_kernel_param结构体, 里面记录字符串密令和func两种数据

struct obs_kernel_param {
    const char *str;
    int (*setup_func)(char *);
    int early;
};

这部分的命令在uboot参数解析的时候, 比如函数

static int __init obsolete_checksetup(char *line);
static int __init do_early_param(char *param, char *val);

这两个函数的不同在于处理的阶段不同, do_early_param函数是在早期的处理里面处理标记为 early 的参数

inode节点

fs\inode.c

对应一个文件, 记录在磁盘的一个inode节点里面, 节点里面记录文件使用的数据块的索引

inode节点存储在硬盘里面, 但是在内存里面保留一个全局数组, 实际使用的是这个数组, 加载硬盘里面的数据暂存在数组里面

  • iget获得一个inode节点
  • iput释放一个inode节点
  • bmap: 对文件进行磁盘映射

inode在硬盘里面记录的是d_inode, 内存里面是m_inode

文件块操作

使用的底层函数是_bmap函数, 对一个文件进行块的获取和释放

/**
 * @brief 块映射处理,为指定i节点的指定块号分配逻辑块
 * 
 * @param inode i节点
 * @param block 数据块号
 * @param create 创建标志，如果置位，则在对应逻辑块不存在时就申请新的磁盘块
 * @return int 数据块对应在设备上的逻辑块号
 */
static int _bmap(struct m_inode * inode,int block,int create);

在inode节点里面有一个i_zone里面记录的是这个节点的磁盘映射, 可以进行新的逻辑块的分配, i_zone里面记录的有7个表项, 一个一级表的位置, 一个二级表的位置, 最大是7+512+512*512个块

这个函数就是解析的函数, 返回的是inode节点的某个数据块对应在设备上的逻辑块号

查找到这个块不存在的时候, 使用函数new_block进行创建

iget/iput获取释放

iget和iput实际都是在操作内存里面的inode节点, 这个节点在同步的时候写入硬盘里面

从硬盘加载使用使用的是read_inode函数, 写入硬盘使用的是write_inode(操作的是硬盘的缓冲区)

在内存的inode表里面预先分配一个空闲 i 节点缓冲位，防止后续死锁, 如果这个表项里面的数据不是空的, 并且更改过了, 对这个表项进行写回(get_empty_inode), 之后检查一下这个inode对应的表项是不是已经被加载了, 是的话处理一下两种

  • 若该 i 节点标记为挂载点, 用被挂载设备号 s_dev 和根节点号 ROOT_INO 替换原来的 dev/nr，重新从表头扫描, 释放掉原挂载点 i 节点的引用，防止泄漏
  • 如果在缓冲表中找到 i 节点且不是挂载点，释放预分配的空闲位，直接返回该 i 节点

若整个缓冲表都未匹配到，则用预分配的空闲位加载硬盘数据

所有的设备挂载在struct super_block super_block[NR_SUPER];这个结构体里面, 最多八个, 使用变量int ROOT_DEV = 0;记录根文件系统的设备号

  • get_super和put_super: 获取某个设备的超级块结构体, 查询数组里面的dev, 设置正在使用, 释放的时候不可以释放根设备, 或者被安装在某个i节点的设备, 释放一下两个位图的映射
  • sys_umount和sys_mount挂载卸载, 卸载的时候依据名字获取inode节点(比如/dev/hda1文件), 检查inode节点数组有没有这个设备里面的节点, 同步设备的缓冲区, 挂载的时候挂载的文件必须是一个目录, 之后就是设置标志位
  • read_super: 从磁盘里面加载, 读取对应的数据, 加载位图到缓冲区里面

Mount原理

  • super_block[i].s_imount：指向挂载点所在原文件系统的 i 节点（目录节点）
  • super_block[i].s_isup：指向被挂载文件系统的根 i 节点
  • m_inode.i_mount：标志该 i 节点是挂载点

sys_mknod 的作用（创建设备文件）：

  • 创建一个特殊文件 i 节点（块设备或字符设备）
  • 设置 i_mode = S_IFBLK（块设备标志）
  • 将设备号存储在 i_zone[0] 中

执行流程示例：

mknod /dev/hda1 b 3 1    ← sys_mknod: 创建块设备文件，i_zone[0] = 0x0301 (主设备3，次设备1)
mount /dev/hda1 /mnt     ← sys_mount: 读 /dev/hda1 的 i 节点 → 取 i_zone[0] = 0x0301 → 用此设备号挂载

关键点：

  • mknod 先行：必须先用 mknod 创建设备文件及其 i 节点
  • mount 依赖 i_zone[0]：mount 通过 i_zone[0] 这个复用字段获得设备号，而不是直接接收设备号参数

Mount 执行流程

  1. 获取设备号：通过设备文件名 dev_name 找到对应 i 节点，从中提取设备号
  2. 获取挂载点：通过目录路径 dir_name 找到目录 i 节点（挂载点），检查其引用计数为 1 且不是根目录
  3. 读取被挂载文件系统的超级块：通过设备号从磁盘读取超级块
  4. 建立关联
    • sb->s_imount = dir_i：超级块记录挂载点 i 节点
    • dir_i->i_mount = 1：i 节点标记为挂载点
