sysfs

2026-04-14
编辑本页

Documentation\core-api\kobject.rst

Linux 统一内核对象模型（kobject/kset）的用户空间视图

它可以产生一个包括所有系统硬件的层级视图, 把连接在系统上的设备和总线组织成为一个分级的文件，它们可以由用户空间存取，向 用户空间导出内核数据结构以及它们的属性

一个Device可以同时存在/device, /class, /bus文件夹下面

不同的类型可以配置不同的attribute, 作为配置参数, 显示在对应的文件夹下面

基础结构

基础文件

  • block: 所有的块设备硬盘、U 盘、分区（sda、sda1、nvme0n1 等）
  • class: 按照功能以及类型进行分类, net（网卡）、input（鼠标键盘）、tty（终端）
  • device: 所有设备实际的真实目录
  • bus: 安装总线进行区分, pci、usb、i2c、spi、platform 等
  • dev: 安装设备号进行索引所有设备
  • fs: 文件系统相关信息和可调参数
  • kernel: 内核运行的时候的信息, 可以调节的参数, 锁, 调度, 打印等
  • module: 所有已经加载的模块
  • power: 电源设备

基础类型

include\linux\kobject.h

kobj

kobject 的目的是抽象出“基类操作”。例如各类设备，都需要做引用计数，都需要生成文件节点，都需要维护文件层级关系。那么就没必要每个设备结构都自己实现一套这些方法，而是每个设备都包含一个kobject就行了，由kobject模块来实现这些方法

  • 每个kset都内嵌了一个kobject，以及拥有一个list；
  • 每个kobject通常都有一个parent object，并且挂在某个kset的list上（kset内嵌的object也可以挂在另一个kset上）；
  • 每个kobject都会指定一个ktype（包括kset内嵌的kobject
struct kobject {
    const char		*name; // 名称
    struct list_head	entry; // 在kset中的链表节点
    struct kobject		*parent; // 父对象的kobject
    struct kset		*kset; // 所属的kset
    struct kobj_type	*ktype; // kobject的类型，包含了释放函数、sysfs操作等
    struct kernfs_node	*sd; /* sysfs directory entry */ // sysfs目录条目
    struct kref		kref; // 引用计数
#ifdef CONFIG_DEBUG_KOBJECT_RELEASE
    struct delayed_work	release;
#endif
    unsigned int state_initialized:1; // 是否已初始化
    unsigned int state_in_sysfs:1; // 是否已在sysfs中注册
    unsigned int state_add_uevent_sent:1; // 是否已发送添加事件的uevent
    unsigned int state_remove_uevent_sent:1; // 是否已发送移除事件的uevent
    unsigned int uevent_suppress:1; // 是否抑制uevent的发送

    ANDROID_KABI_RESERVE(1); // 保留字段，供未来扩展使用
    ANDROID_KABI_RESERVE(2);
    ANDROID_KABI_RESERVE(3);
    ANDROID_KABI_RESERVE(4);
};

本身一般不是关注重点；它通常嵌入在其他结构体中，后者才是真正的业务对象。

ktype

ktype嵌入 kobject 的对象类型。每个嵌入 kobject 的结构体都需要一个对应的 ktype。ktype 控制该 kobject 在创建和销毁时的行为。

kset

kset：kobject 的集合。集合中的对象可以是同一种 ktype，也可以不是。kset 是对象集合的基本容器。kset 自身也包含一个 kobject，但通常可以忽略这个实现细节，核心代码会处理它。

当你在 sysfs 中看到一个目录下有很多子目录时，通常这些子目录对应同一个 kset 中的各个 kobject。

  • 作为对象集合容器（例如“所有块设备”“所有 PCI 驱动”）
  • 作为 sysfs 子目录，集合成员可出现在其中
  • 支持对象热插拔事件（uevent）并影响上报行为

初始化

lib\kobject.c

kobj

  • kobject_init: 初始化一个konject的结构体, 初始化基础的数据信息以及赋值ktype
  • kobject_add: 注册函数, 记录这个kobject到sysfs里面, 使用格式化字符串初始化名字若 kobject 要关联某个 kset，需在调用前先给 kobj->kset 赋值。
    若已关联 kset，调用 kobject_add() 时 parent 可以是 NULL，此时父对象会设为该 kset。

可以直接使用函数kobject_init_and_add一步实现

kset

由于 kset 内含 kobject，也应动态创建, 可以使用函数kset_create_and_add

uevent事件

kobject 成功注册后，还需要通知外部它已创建, 通过函数kobject_uevent发送

新增对象应使用 KOBJ_ADD。
应在属性与子对象都初始化完成后再发，否则用户态会立刻开始访问，可能看到不完整状态。

对象移除时，KOBJ_REMOVE 事件由 kobject 核心自动产生，调用方无需手工发送。

引用计数

kobject 的核心作用之一是为其宿主对象提供引用计数。只要还有引用，对象及其支撑代码就必须继续存在。相关底层接口：

  • struct kobject *kobject_get(struct kobject *kobj);
  • void kobject_put(struct kobject *kobj);

kobject_get() 成功会增加引用计数并返回指针。
kobject_put() 会减少引用计数，并可能在计数归零时释放对象。

创建方通常不知道“最后一次释放”何时发生（知道的话用引用计数就意义不大了）

“最后一个引用消失”的异步通知通过 release() 回调完成。典型形式：

  • 从 struct kobject * 用 container_of() 回到宿主结构
  • 做额外清理
  • 最后 kfree() 宿主对象

最重要的一点：每个 kobject 都必须有 release()，并且对象在回调前必须保持一致可用。否则设计有缺陷。内核会在缺少 release() 时报警。

访问

sysfs 本质是基于 Linux 内核的 kernfs 框架实现的，而 kernfs_node 是 kernfs 的核心数据结构，inode/dentry 则是文件系统通用层与 kernfs 交互的桥梁

用户看到的 /sys/xxx 下的每一个目录、文件、软链接，在内核里都对应一个 kernfs_node 结构体实例

kernfs 是 kernel 3.14 引入的通用虚拟文件系统框架（而非面向用户的独立文件系统）。

它封装了虚拟文件系统的通用逻辑（目录树管理、节点生命周期、inode/dentry 关联、权限控制、红黑树查找等），让上层虚拟文件系统（如 sysfs、configfs）无需重复造轮子，只需聚焦自身业务逻辑

sysfs_init 通过 kernfs_create_root(struct kernfs_syscall_ops *scops, unsigned int flags, void *priv)一个新的 kernfs 层次，然后将其保存在静态全局变量 sysfs_root中，供各处使用。然后通过register_filesystem将其注册为名为 sysfs 的文件系统。

结构名 核心作用
kernfs_node 最核心：每个实例对应一个目录 / 文件 / 软链接，存储名称、父节点、ID、类型、权限、私有数据等
kernfs_iattrs 存储 inode 扩展属性（i_atime/i_mtime/i_uid/i_gid），复用减少内存占用
kernfs_ops 节点操作方法集：定义读 / 写 / 查找 / 删除等回调，上层 FS 可重载实现自定义逻辑
kernfs_idr ID 映射器：维护 kernfs_node.id ↔ kernfs_node 的映射，关联 inode 的 i_ino
kernfs_elem_* 联合体（dir/symlink/attr）：按节点类型存储专属数据（如目录的子节点红黑树根、文件的读写缓存）
类型 核心接口 作用
节点创建 kernfs_create_dir() 创建目录节点
kernfs_create_file() 创建文件节点（如 sysfs 的属性文件）
kernfs_create_symlink() 创建软链接节点
节点销毁 kernfs_remove() 销毁指定节点
kernfs_unlink() 从父节点中移除子节点
节点查找 kernfs_find_and_get() 按名称查找目录下的子节点
kernfs_get_parent() 获取节点的父节点

kernfs_getattr() 获取节点的属性（同步到 inode）

kernfs_node 的核心作用

  • 作为 sysfs 节点（目录 / 文件）的内核唯一标识，存储节点的名称、类型、权限、父 / 子节点关系；
  • 关联 VFS 层的 inode（文件系统通用索引节点），是 kernfs 与 VFS 交互的核心桥梁；
  • 绑定 sysfs 节点的操作方法（如读 / 写 / 创建 / 删除）；
  • 关联内核对象（如 kobject），实现「内核对象 ↔ sysfs 节点」的映射。
/*
 * kernfs_node - the building block of kernfs hierarchy.  Each and every
 * kernfs node is represented by single kernfs_node.  Most fields are
 * private to kernfs and shouldn't be accessed directly by kernfs users.
 *
 * As long as s_count reference is held, the kernfs_node itself is
 * accessible.  Dereferencing elem or any other outer entity requires
 * active reference.
 */
struct kernfs_node {
    atomic_t		count;
    atomic_t		active;
#ifdef CONFIG_DEBUG_LOCK_ALLOC
    struct lockdep_map	dep_map;
#endif
    /*
     * Use kernfs_get_parent() and kernfs_name/path() instead of
     * accessing the following two fields directly.  If the node is
     * never moved to a different parent, it is safe to access the
     * parent directly.
     */
    struct kernfs_node	*parent;
    const char		*name; // 节点名称（文件名 / 目录名）
    // 红黑树节点，kernfs 用红黑树管理同一目录下的所有子节点
    struct rb_node		rb;

    const void		*ns;	/* namespace tag */
    unsigned int		hash;	/* ns + name hash */
    // 按节点类型存储专属数据
    union {
    	struct kernfs_elem_dir		dir;
    	struct kernfs_elem_symlink	symlink;
    	struct kernfs_elem_attr		attr;
    };

    void			*priv; // 私有数据（通常指向 kobject）

    /*
     * 64bit unique ID.  On 64bit ino setups, id is the ino.  On 32bit,
     * the low 32bits are ino and upper generation.
     */
    u64			id; // 直接对应 VFS inode 的 i_ino（索引节点号）

    unsigned short		flags;
    umode_t			mode; // 决定 inode 的类型（目录 / 文件 / 软链接）
    // 存储 inode 的扩展属性, 创建时间用户组等
    struct kernfs_iattrs	*iattr;

    ANDROID_KABI_RESERVE(1);
};

当调用 kernfs_create_dir()/kernfs_create_file() 创建 sysfs 节点时：

  1. 内核为新的 kernfs_node 分配一个全局唯一的 id（由 kernfs_idr 这个 ID 映射器管理）；
  2. 这个 id 会作为后续 inode 的 i_ino（索引节点号）—— 这是两者的核心绑定点。

链接建立

sysfs 作为 VFS 子系统，当需要为 kernfs_node 创建 inode 时（比如用户首次访问 /sys/xxx 路径）：

  1. 内核调用 sysfs_get_inode()（kernfs 专属的 inode 分配函数）；
  2. 从 kernfs_idr 中根据 id 找到对应的 kernfs_node；
  3. 填充 inode 的核心字段：
    • i_ino = kernfs_node.id（绑定 ID）；
    • i_mode = kernfs_node.mode（权限同步）；
    • i_op = sysfs_inode_operations（sysfs 专属的 inode 操作方法）；
    • i_fop = 对应节点类型的文件操作方法（如目录的 sysfs_dir_operations）；
    • i_private = kernfs_node（反向关联核心：让 inode 能直接找到对应的 kernfs_node）；
    • i_atime/i_mtime = kernfs_node.iattr 中的时间属性；
  4. 此时，kernfs_node ↔ inode的关联关系就建立了：
    • 正向：kernfs_node.id → inode.i_ino（通过 kernfs_idr 映射）；
    • 反向：inode.i_private → kernfs_node（直接指针）。

  • 属性文件的创建

“属性” 在 sysfs 中对应的是文件(file)。当需要为设备添加属性时，可以调用 device_create_file ，于是：

device_create_file => sysfs_create_file => sysfs_create_file_ns => sysfs_add_file_mode_ns => __kernfs_create_file 为属性创建文件，并根据 kobj->ktype->sysfs_ops 为文件绑定对应的读写函数。

于是在读文件时，调用 sysfs_kf_read ，它会根据属性文件找到其父节点类型对应的 sysfs_ops ，然后调用 sysfs_ops.show 。show 需要将输出写到传入的 buf 缓冲区中，并返回写入的长度。

在写文件时，调用 sysfs_kf_write ，它会根据属性文件找到其父节点类型对应的 sysfs_ops ，然后调用 sysfs_ops.store 。 store 可以从传入的 buf 缓冲区中，读取用户写入的长度为 len 的内容。
