内存屏障

用于控制内存访问顺序的技术。它主要用于多处理器系统中，确保某些操作按预期顺序执行，避免 CPU 和编译器对内存访问进行优化，从而影响程序的正确性。它的作用是禁止 CPU 重新排序特定的内存操作。它确保在屏障之前的所有读/写操作在屏障之后的操作之前完成。

  1. 全屏障（Full Barrier 或者 LFENCE、SFENCE）：
  • 作用：以下两种相加。
  • 用途：确保所有的内存操作都在内存屏障前完成，通常用于同步和锁定操作。
  • 内核函数：mb()(Memory Barrier)

2.读屏障（Read Barrier 或者 LFENCE）：

  • 作用：保证屏障之前的所有读操作在屏障之后的读操作之前完成。---》“完成”是缓存同步主内存的意思。
  • 本质：作用是强制将 CPU核心 中的 L1/L2 缓存 中的共享变量值写回到 主内存。
  • 用途：在执行并行读操作时确保读顺序。
  • 内核函数：rmb() (Read Memory Barrier)

3.写屏障（Write Barrier 或者 SFENCE）：

  • 作用：保证屏障之前的所有写操作在屏障之后的写操作之前完成。--》“完成”是主内存同步到缓存的意思。
  • 本质：作用是强制使数据从主内存加载，而不是直接使用可能已经过时的缓存数据。
  • 用途：用于确保写操作顺序。
  • 内核函数：wmb() (Write Memory Barrier)

4.无序屏障（No-op Barrier）：

  • 作用：没有实际影响，仅确保 CPU 不会重排序特定的指令。
  • 用途：常用于确保指令的顺序性而不做其他强制性的内存同步。

原子操作

原子整型操作接口

原子操作的各种实现实际是更具不同的架构是不同的

原子整型变量

typedef struct {
int counter;
} atomic_t;
ATOMIC_INIT()

定义整型原子变量并设置初始值

#define ATOMIC_INIT(i) { (i) }
atomic_t data = ATOMIC_INIT(int i)

WRITE_ONCE 宏保证这个写操作不会被编译器优化或重排序

atomic_set()

用于设置整型原子变量的值

atomic_set(atomic_t *v,int i)

这个操作并不是原子操作, 只是使用volatile保证确实是操作到这个数据

atomic_read()

获取原子变量的值

#define atomic_add(i,v)			(void)atomic_add_return((i), (v))
atomic_read(atomic_t *v)
atomic_add()/atomic_sub()

整型原子变量的加/减

static inline void atomic_add(int i, atomic_t *v) //整型原子变量加i
static inline void atomic_sub(int i, atomic_t *v) //整型原子变量减i
atomic_inc()/atomic_dec()

整型原子变量自增/自减

static inline void atomic_inc(atomic_t *v); //整型原子变量自增1
static inline void atomic_dec(atomic_t *v); //整型原子变量自减1
// 返回成功以后得数值
static __always_inline int atomic_inc_return(atomic_t *v);
static __always_inline int atomic_dec_return(atomic_t *v);
// 判断是不是 == 0
static __always_inline bool atomic_inc_and_test(atomic_t *v);
static __always_inline bool atomic_dec_and_test(atomic_t *v);

atomic_xchg()

它会把你传入的新值（如 0 或 1）写入 atomic_t 变量，并返回原来的旧值。整个过程是原子的，保证不会被中断或并发修改

static __always_inline int atomic_xchg(atomic_t *v, int i);

位原子操作函数

set_bit()

将某一位置1

set_bit(int nr, unsigned long *addr)
clear_bit()

将某一位置0

clear_bit(int nr, unsigned long *addr)
change_bit()

反转某一位的值

int test_bit(int nr, unsigned long \*addr)

static atomic_t test_atomic = ATOMIC_INIT(1);

/*字符设备操作函数集，open函数*/
static int led_chr_dev_open(struct inode *inode, struct file *filp)
{
    if (atomic_xchg(&test_atomic, 0) == 1) {
        printk("\n open from driver \n");
        return 0;
    } else {
        printk("\n driver on using! open failed !!!\n");
        return -EBUSY;
    }
}


static int led_chrdev_release(struct inode *inode, struct file *filp)
{
    atomic_set(&test_atomic,1);
    printk("KERN_ALERT  \n finished  !!!\n");

    return 0;
}

自旋锁/读写锁/顺序锁

include\linux\spinlock.h进行实现, 但具体实现依赖于架构, arch/arm64/include/asm/spinlock.h

  • ARM64 自旋锁本质是一个 32 位整数，0 表示未加锁，1 表示已加锁。
  • 加锁用原子指令循环尝试设置为1，解锁直接原子写0。
  • 在等待锁的时候不会调度, 而是进行死锁, 为了避免优先级翻转, 获取自旋锁时，内核会自动禁用当前 CPU 的抢占（preempt_disable ()）；释放锁时，自动启用抢占（preempt_enable ()）

在临界区里面不可以使用会导致睡眠以及任务切换的函数

在实际使用的时候, 中断里面尝试获取自旋锁会出现问题, 所以实际的应用的时候可可以关闭中断, 还有针对于下半部的关闭函数

在实际应用的时候有一个情况是可以多个进行读取, 但是只有一个可以改, 延伸出来读写锁

顺序锁在读写锁的基础上衍生而来的，使用读写锁的时候读操作和写操作不能同时进行。使用顺序锁的话可以允许在写的时候进行读操作，也就是实现同时读写，但是不允许同时进行并发的写操作。

但是如果在读的过程中发生了写操作，最好重新进行读取，保证数据完整性。顺序锁保护的资源不能是指针，因为如果在写操作的时候可能会导致指针无效

seqlock（顺序锁）允许“多读单写”，但不是“读写完全同时进行”。它的机制是：

  • 多个读者可以并发、无锁地读取数据（只要没有写者在写）。
  • 写者写数据时，读者会检测到有写操作（通过序列号变为奇数），此时读者会等待（自旋）直到写者完成，或者重试读取。
  • 写者之间是互斥的，不能同时写。
  • 读者和写者不能真正“同时”访问同一份数据：如果读者在写者写入期间读取，读者会检测到数据不一致，自动重试，直到读到一致快照。

所以，seqlock 的本质是“读者无锁、写者加锁”，但写期间读者会自旋等待或重试，保证数据一致性。它适合读多写少、对偶尔重试能容忍的场景。

do {
  seqnum = read_seqbegin(&seqlock_a);
  /* 读操作代码块 */
  ...
} while (read
_seqretry(&seqlock_a, seqnum));

自旋锁

主要用于多核处理器, 短时间轻量级的等待, 被自旋锁保护的临界区一定不能调用任何能够引起睡眠和阻塞的

API 函数

  • 短时期的轻量级加锁

  • 加锁失败时，原地打转、忙等待

  • 避免上下文调度、系统开销较小

加锁步骤：

  1. 查看锁的状态，如果锁是空闲的
  2. 将锁设置为当前线程持有

在没有 CAS 函数前，多个线程同时执行这 2 个步骤是会出错的。

CAS 函数把这 2 个步骤合并为一条硬件级指令。这样第 1 步比较锁状态和第 2 步锁变量赋值，将变为不可分割的原子指令(硬件同步原语)

自旋锁使用CPU 提供的 CAS 函数（Compare And Swap），在用户态代码中完成加锁与解锁操作

自旋锁并不一直"忙等待"，会与 CPU紧密配合 ，它通过 CPU 提供的 PAUSE 指令，减少循环等待时的耗电量；对于单核CPU，忙等待并没有意义，此时它会主动把线程休眠。

RCU

通常用于链表操作, 用于保护里面的数据 Documentation\RCU\whatisRCU.rst

RCU（Read-Copy-Update，读 - 复制 - 更新）是 Linux 内核中一种读无锁、写延迟回收的高性能同步机制，专为读多写少场景设计，核心是让读操作完全无锁、零开销，写操作通过复制修改 + 原子替换 + 宽限期回收三步实现并发安全

a. 移除对数据结构的指针，使后续读者无法再引用它。
b. 等待所有之前的读者完成RCU读侧临界区。
c. 此时没有读者再引用该数据结构，可以安全回收（如kfree）。

读端临界区（Read-Side Critical Section）

  • 用 rcu_read_lock() 标记开始，rcu_read_unlock() 标记结束。
  • 作用：告知内核当前处于 RCU 读状态，禁止抢占（普通 RCU），保证读期间数据指针不被回收Linux Kernel。
  • 约束：临界区内不能睡眠、不能阻塞、不能进入用户态 /idle。

宽限期

定义：写操作完成指针替换后，必须等待的所有 CPU 都退出旧数据读临界区的最短时间。

判断：所有 CPU 至少完成一次上下文切换 / 用户态执行 /idle 循环，即所有旧读者已退出Linux Kernel。宽限期结束后，旧数据无任何读者引用，可安全释放

写端流程（Read-Copy-Update）

  • 复制：写者不修改原数据，先复制一份副本并修改。
  • 原子替换：用原子操作（如 rcu_assign_pointer）将全局指针指向新副本，后续读者读到新数据。
  • 等待宽限期：调用 synchronize_rcu()（阻塞等待）或 call_rcu()（注册回调异步执行）。
  • 回收旧数据：宽限期结束后，释放旧数据内存（如 kfree_rcu）。

常见的应用形式是链表

示例

  • 链表
// 定义RCU保护的链表
struct foo {
    int data;
    struct list_head list;
};
LIST_HEAD(g_foo_list);

// 读操作（无锁）
void foo_read(void) {
    struct foo *fp;
    rcu_read_lock(); // 进入读临界区
    list_for_each_entry_rcu(fp, &g_foo_list, list) {
        printk("%d\n", fp->data); // 安全访问
    }
    rcu_read_unlock(); // 退出读临界区
}

// 写操作（复制+替换+回收）

DEFINE_SPINLOCK(g_foo_lock);
void foo_update(int old_data, int new_data) {
    struct foo *new_fp, *old_fp;
    // 1. 复制并修改
    new_fp = kmalloc(sizeof(*new_fp), GFP_KERNEL);
    new_fp->data = new_data;
    // 2. 原子替换, 使用锁避免写竞争
    spin_lock(&g_foo_lock);
    rcu_read_lock();
    list_for_each_entry_rcu(old_fp, &g_foo_list, list) {
        if (old_fp->data == old_data) {
            // 标记需要替换
            list_replace_rcu(&old_fp->list, &new_fp->list);
            break;
        }
    }
    rcu_read_unlock();
    spin_unlock(&g_foo_lock);
    // 3. 等待宽限期并回收旧数据
    synchronize_rcu();
    kfree(old_fp);
}
  • 基础数据
// 全局指针，用RCU保护
struct config *g_config __rcu;

// 读操作（无锁）
void read_config(void) {
    struct config *cfg;
    rcu_read_lock();
    cfg = rcu_dereference(g_config); // 安全读取指针
    if (cfg) {
        printk("config: %d\n", cfg->timeout);
    }
    rcu_read_unlock();
}

// 写操作（复制-替换-回收）
void update_config(int new_timeout) {
    struct config *new_cfg = kmalloc(sizeof(*new_cfg), GFP_KERNEL);
    struct config *old_cfg;
    new_cfg->timeout = new_timeout;

    // 原子替换指针
    old_cfg = rcu_replace_pointer(g_config, new_cfg, GFP_KERNEL);
    // 等待宽限期后回收旧配置
    synchronize_rcu();
    kfree(old_cfg);
}

__rcu 定义在内核头文件 <linux/rcupdate.h> 中，本质是给指针添加的__attribute__属性（不同架构 / 编译器会适配），核心定义简化后如下：

// 核心定义（简化版）
#define __rcu __attribute__((noderef, address_space(4)))
  • noderef：提示静态检查工具sparse—— 不能直接解引用这个指针（必须通过 RCU 接口）；
  • address_space(4)：标记指针属于 RCU 地址空间，仅用于检查。

completion是一种代码同步机制，比锁/信号量更可取。如果有一个或多个线程必须等待某个内核活动达到某个点或某个特定状态，那么completion完成量可以为这个问题提供一个无竞争的解决方案。

总的来说，使用完成量需要三个步骤：

（1）初始化struct completion同步对象。

（2）通过调用wait_for_completion()函数等待完成量。

（3）调用complete()或complete_all发送完成信号。

#define init_completion(x) __init_completion(x)
void wait_for_completion(struct completion *done);
void complete(struct completion *x);
void complete_all(struct completion *x);
